# 광고 도입 — 운영 핸드오프 (사용자가 직접 처리할 일)

> 대기 시간 광고 슬롯 기능(`docs/광고-도입-기획.md`)의 **남은 일** 중, 코드로 끝낼 수 없고 **비즈니스·운영·법무 결정 또는 외부 계정**이 필요해 사용자가 직접 처리해야 하는 항목을 정리한다.
> 코드 쪽에서 끝낸 부분(슬롯 인프라·게이팅·플레이스홀더·on/off 플래그·계측 리포터 seam)은 본 문서 §5 "코드 확장 지점"에 어디를 건드리면 되는지까지 표로 남겼다.

## 1. 분석 싱크(노출/클릭 계측)의 목적지 결정·연결

**왜 사용자 몫인가:** 코드에는 노출(impression)·클릭 이벤트를 발생시키는 지점과 교체 가능한 리포터 인터페이스가 이미 배선돼 있다. 다만 그 이벤트를 **어디에 영구 저장·집계할지**는 제품/인프라 결정이다(자체 백엔드 vs 외부 분석 도구). 현재 기본 구현은 브라우저 콘솔에 찍는 **개발/QA용 임시 싱크**라, KPI("노출 계측 커버리지 100%")를 실제로 채우지는 못한다.

**결정해야 할 것 (택1 또는 혼합):**

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| (a) 자체 백엔드 엔드포인트(`POST /ad-events`) | 데이터 소유·PII 통제 쉬움, 기존 스택(Spring) 재사용 | 서버/DB 작업 필요 |
| (b) 외부 분석 도구(GA4·Amplitude 등) | 빠른 도입·대시보드 무료 | 외부 추적 스크립트→개인정보·동의 이슈, Canvas 환경 연동 품 |
| (c) 당분간 콘솔/로그만 | 추가 작업 0 | KPI 측정 불가(임시) |

**권고:** 1차 KPI가 "노출/완주율 비열등성"뿐이라면 **(a) 최소 엔드포인트**가 데이터 통제·개인정보 측면에서 가장 깔끔하다. 이벤트 스키마는 비식별(노출/클릭 카운트 + 광고 식별자 + 타임스탬프)로 한정한다(기획 NFR-6).

**연결 방법(코드):** `AdMetricsReporter`(§5)를 구현한 새 클래스(예: `HttpAdMetricsReporter`)를 만들어 `App.kt`에서 `ConsoleAdMetricsReporter` 대신 주입하면 된다. **슬롯·게이팅·화면 코드는 한 줄도 바꿀 필요 없다.**

## 2. Baseline(광고 도입 전) 완주율 측정 — 비열등성 판정의 선행조건

**왜 사용자 몫인가:** 기획 §10.2는 "광고가 완주율을 3%p 넘게 떨어뜨리지 않는다(비열등성)"를 출시 게이트로 못 박았다. 이 판정은 **광고 없는 기간의 기준선**이 있어야 가능하다. 코드는 측정을 자동화하지 않는다 — 데이터 수집·기간 운영은 사람이 한다.

**필요 선행:** §1의 분석 싱크(완주 이벤트=제출→완성 열람 집계)가 먼저 있어야 한다.

**런북(절차):**
1. **광고 OFF로 배포** — 배포 환경변수 `ADS_ENABLED=false`(§5 플래그). 이 상태로 **2주 이상** 운영하며 완주율을 모은다(기준선 A).
2. **광고 ON으로 전환** — `ADS_ENABLED=true`. 같은 방식으로 완주율을 모은다(처치군 B).
3. **표본 확보** — 각 군 **완주 이벤트 300건 이상**.
4. **판정** — B의 완주율이 A 대비 **3%p 초과 하락하지 않으면 PASS**(비열등성). 초과하면 배치·빈도를 재검토하거나 광고를 OFF로 되돌린다(킬스위치).

> 참고: A/B를 동시 분할(같은 기간 절반은 광고, 절반은 무광고)하면 계절성 편향이 줄지만, 그러려면 사용자 단위 분기 로직이 추가로 필요하다(현재 플래그는 전역 on/off). 1차는 위 순차 비교로 충분하다고 본다.

## 3. 실제 광고망(AdSense 등) 연동

**왜 사용자 몫인가:** 외부 광고망은 **계정 개설·도메인/퍼블리셔 승인·정책 심사**가 선행돼야 하고, 운영 HTTPS·콘텐츠 정책 충족이 필요하다. 코드 샌드박스에서 실광고 검증도 불가하다.

**처리 순서(권고):**
1. **계정·승인** — 광고망 가입, 도메인 등록, 퍼블리셔 ID 발급, 정책 심사 통과.
2. **법무 확인** — §4 먼저.
3. **기술 연동** — 이 프로젝트 프런트는 **Compose Wasm/Canvas**라 화면 전체가 단일 캔버스다. DOM 기반 광고 스크립트는 **캔버스 위 HTML 오버레이 레이어**로 띄워야 한다. 연동은 다음 둘 중 하나로 캡슐화한다:
   - `AdContentProvider` 구현체가 캔버스 위 오버레이 div를 만들어 광고 스크립트를 적재(슬롯 위치는 화면에서 계산해 전달).
   - 또는 광고 영역만 별도 DOM 컨테이너로 분리(`index.html`).
   어느 쪽이든 **슬롯/게이팅/라벨/정당화 카피 코드는 재사용**되고, 바뀌는 건 "슬롯 안에 무엇을 그리느냐"(콘텐츠 공급자)뿐이다.
4. **라벨·표기 유지** — 실광고로 바꿔도 "광고" 라벨과 정당화 카피, 실패-미노출·완성-즉시철거 정책은 유지한다(기획 4기둥).

## 4. 법무 확인 (광고 표기·개인정보)

**왜 사용자 몫인가:** 법적 판단이며 서비스 도메인·대상에 따라 다르다.

**확인 항목:**
- **광고 표기 의무** — 국내 인터넷 광고 관련 법규상 "광고"임을 명확히 표기해야 한다. 현재 자사 플레이스홀더 단계는 "광고" 배지로 충족하나, **실광고·제휴 도입 시 표기 요건(광고 주체·"광고"/"Sponsored" 등)을 법무와 재확인**.
- **개인정보·추적 동의** — 실광고망/외부 분석은 쿠키·추적 식별자를 쓸 수 있다. 동의(consent) 배너·개인정보처리방침 갱신 필요 여부를 확인. (1차 자사 플레이스홀더는 외부 추적·PII 수집이 없어 해당 없음 — 기획 NFR-6.)
- **결제·수익 신고** — 광고 수익 발생 시 사업자/세무 처리(기획의 결제 보류와 별개 트랙).

## 5. 코드 확장 지점 요약 (개발자가 후속 작업 시 어디를 건드리나)

| 하려는 일 | 건드릴 곳 | 안 건드려도 되는 곳 |
| --- | --- | --- |
| 실제 분석 싱크 연결 | `AdMetricsReporter` 구현 새 클래스 + `App.kt`에서 주입 교체 | 슬롯·게이팅·화면 |
| 실제 광고 콘텐츠로 교체 | `AdContentProvider` 구현 새 클래스(어댑터) + `App.kt`(또는 `ArtifactListScreen` `adContentProvider` 인자) 주입 | `AdSlot`·게이팅·라벨·카피 |
| 광고 끄기/켜기(운영) | 배포 환경변수 `ADS_ENABLED`(아래) — 코드 변경 0 | 전부 |
| 노출 위치 추가(2차 자리) | `AdSlot` 재사용 + 새 게이팅 조건 주입 | 슬롯 내부 |

**광고 on/off 플래그 운영 레퍼런스:**
- 런타임 전역: `window.__RESUMAKER_ADS_ENABLED__` (문자열 `"true"`/`"false"`). **명시적 `"false"`일 때만 OFF**, 미설정·그 외는 ON.
- 로컬 dev: `app/webApp/src/webMain/resources/config.js`에서 직접 수정.
- 배포(nginx 컨테이너): 환경변수 `ADS_ENABLED`로 주입(`app/webApp/nginx/40-api-base.sh`가 `config.js`에 굽는다). **재빌드 없이** 환경변수만 바꿔 끄고 켤 수 있다 → baseline 측정·킬스위치에 사용.
- 코드 진입점: `watson.resumaker.config.adsEnabled()` (`RuntimeConfig.kt`, expect/actual). `App.kt`가 읽어 `ArtifactListScreen(adsEnabled = ...)`로 전달.

**ads.txt·소유권 메타·콘텐츠 URL 운영 레퍼런스:**
- **자동 생성됨**: 프런트 컨테이너가 뜰 때마다 `40-api-base.sh`(`/docker-entrypoint.d/`)가 환경변수로 ads.txt·소유권 확인 메타 태그·sitemap/robots/canonical을 굽는다. **이미지 빌드가 아니라 런타임**이라 환경변수만 바꿔 컨테이너 재기동하면 반영(재빌드 불필요).
- `ADSENSE_CLIENT`(ca-pub-…): **설정 시** `/ads.txt`에 `google.com, pub-…, DIRECT, f08c47fec0942fa0`(ca- 제거)와 `<meta name="google-adsense-account">`가 생성. **미설정(현재)** 이면 ads.txt는 주석만(무효 행 없음)·메타 없음 → AdSense 무동작(안전). 발급 후 `.env.prod`에 넣고 재기동하면 자동 유효화.
- `SITE_ORIGIN`(예: `https://app.<도메인>`): canonical·sitemap·robots의 절대 URL 기준. **배포 전 반드시 운영 값으로 지정**(미지정 시 기본 `http://localhost:8081`이 그대로 박혀 콘텐츠 URL이 깨짐).
- ⚠️ **루트→app 리다이렉트**: AdSense는 루트 도메인만 등록되고 Cloudflare가 루트를 `app.<도메인>`으로 보낸다. ads.txt 크롤러는 루트를 보지만 같은 루트 도메인 내부 리다이렉트라 따라가진다. **경로 보존(path-preserving) 301 한 홉** 필수 — `<도메인>/ads.txt`→`app.<도메인>/ads.txt`(끝 경로 유지). 검증: `curl -sI https://<도메인>/ads.txt`의 Location 확인. 상세는 `docs/광고-AdSense-전략.md` §4.2.
