이 문서는 'Resumaker'에서 **"좋은 이력서란 무엇인가"** 를 정의하고, 서비스가 생성한 산출물을 그 기준에 따라 **추가로 자동 개선(Quality Improvement)** 하는 기능을 도메인 관점에서 기획한다. 이 문서는 디자이너·개발자·개발팀장이 추측 없이 구현에 착수할 수 있을 만큼 구체적이어야 한다.

작성 규칙은 기존 도메인 문서를 따른다. 도메인 관점(무엇을·왜)에 집중하고, 도메인 용어는 제목에서 영어를 한 번 노출한 뒤 이후 한글만 쓴다. 각 기능은 목적·이뤄낼 가치·시나리오·수용 기준을 갖춘다.

> 진실의 원천: 이 기획은 `Resumaker 도메인 이해.md`의 결정을 전제로 한다. 특히 §459 신뢰성 가드레일(근거 없는 사실 금지, 표현 다듬기만 허용)이 이 기능의 **가장 강한 제약**이며, 충돌 시 언제나 신뢰성 가드레일이 우선한다. 이 문서는 기존 §308 개선/재생성, §389 산출물 구조와 상태 모델, §277 비동기 생성 작업, §440 비용 가드레일과 정합한다.

# 0. 이 기능이 닫는 페인포인트

| # | 페인포인트 | 이 기능이 어떻게 돕는가 |
|---|-----------|----------------------|
| P4 | **"어디를 어떻게 고쳐야 할지 모르겠다"** — 완성한 초안의 어느 부분이 약한지, 어떻게 개선할지 판단이 어렵다. | 기존 §308은 **사용자가 약점을 스스로 찾아** 항목을 재생성·편집하게 한다. 이 기능은 그 앞단에서 **"어디가 약한지"를 서비스가 먼저 진단**하고, 안전한 범위에서 **다듬어진 후보를 먼저 제시**한다. P4의 "어디를"을 서비스가 답한다. |
| P3 | **"과장·거짓말이 될까 봐 두렵다"** | 자동 개선은 표현·구조·강조·간결화만 손대고 사실(수치·고유명사·성과)을 절대 추가·변경하지 않는다. 개선 결과에도 기존 자동 신뢰성 검증을 그대로 적용한다(아래 §4.4). 따라서 "다듬어도 부풀지 않는다"를 보장한다. |

> 위치 근거: 기존 §308 개선/재생성이 P4의 **수단(재생성·편집·버전)** 을 제공한다면, 이 기능은 P4의 **진단(어디가 약한가)** 과 **무위험 자동 처치(표현 다듬기)** 를 더한다. 즉 §308을 대체하지 않고 그 앞단에 진단·제안 계층을 얹는다.

# 1. 좋은 이력서의 정의 (품질 기준)

이 절은 취업 준비생 맥락에서 "좋은 이력서가 만족하는 조건"을 범주화한다. 각 기준에는 **점검 가능성**을 표기한다.

- **자동(A):** 결정적 규칙(추출·문자열 대조·길이·패턴)으로 서비스가 위반을 판정할 수 있다. 사용자 개입 없이 진단된다.
- **반자동(S):** 휴리스틱(사전 기반·간단한 통계)으로 후보를 표시하되, 최종 판단은 사람(또는 LLM 보조)에 맡긴다. 오탐 가능성을 전제로 "제안"으로만 노출한다.
- **사용자검토(U):** 결정적·반자동 판정이 불가능해, 사용자에게 검토를 위임한다.

## 1.1 진실성·신뢰성 (Truthfulness)
가장 상위 기준. 다른 모든 기준보다 우선한다.

| 기준 | 내용 | 점검 |
|---|---|---|
| T1 정량 수치 근거 | 모든 수치(%, ms, 건수, 금액 등)는 경험 기록에 문자적 근거가 있다. | **자동(A)** — 기존 자동 검증(§473) 재사용 |
| T2 고유명사 근거 | 회사명·기술명·프로젝트명은 경험 기록에 문자적 근거가 있다. | **자동(A)** — 기존 자동 검증 재사용 |
| T3 성과 주장 근거 | "최우수상 수상", "MAU 10만" 등 성과 주장은 근거가 있다. | **사용자검토(U)** — 자유 서술이라 결정적 추출 불가(§479), 생성 지시·검토로 다룸 |

> 자동 개선의 절대 경계: 자동 개선은 T1·T2·T3을 **위반하지 않는 범위에서만** 동작한다. 사실을 추가·변경하지 않으므로 개선이 T1~T3을 새로 위반시킬 수 없어야 하고, 만약 개선 결과가 검증을 통과하지 못하면 그 항목은 채택되지 않는다(§4.4).

## 1.2 목표 적합성 (Target Fit)
| 기준 | 내용 | 점검 |
|---|---|---|
| F1 목표 강조 정렬 | 산출물의 강조점이 목표의 작성 전략(강조 키워드·강조할 점)과 정렬된다. | **반자동(S)** — 작성 전략 키워드의 산출물 내 등장 여부는 자동 점검 가능하나, "잘 강조됐는가"의 질은 사용자검토 |
| F2 목표 무관 내용 배제 | 목표와 무관한 곁가지(작성 전략의 '피할 점')가 전면에 나오지 않는다. | **반자동(S)** — '피할 점' 키워드 등장은 자동 표시, 제거 여부 판단은 사용자 |

## 1.3 임팩트·구체성 (Impact)
| 기준 | 내용 | 점검 |
|---|---|---|
| I1 강한 동사 | 책임 나열("담당했다", "참여했다")이 아니라 행동·성취 동사("개선했다", "구축했다", "단축했다")를 쓴다. | **반자동(S)** — 약한 동사 사전 매칭으로 후보 표시 |
| I2 능동태 | 수동태("개선되었다") 대신 능동태("개선했다")로 주체를 드러낸다. | **반자동(S)** — 수동 표현 패턴 매칭 |
| I3 성과 가시화 | 경험 기록에 결과가 있으면, 그 결과가 산출물 항목에서 드러난다. | **반자동(S)** — 경험의 결과 필드가 비어있지 않은데 항목에 미반영이면 표시. 단 **수치 추가는 금지**(T1) |
| I4 객관적 수치화 (as-is→to-be) | 모호 수치·추상 규모어("200% 증가", "대용량 트래픽", "성능 30% 개선")가 아니라 **대상+실측값+단위**로 무엇이 어떻게 변했는지 명시한다("DAU 100→200명", "평균 500tps", "응답시간 300ms→210ms"). | **반자동(S)** — 모호 수치·규모어 패턴은 자동 표시. 객관 수치로의 치환은 **경험 기록에 그 실측값이 있을 때만** 자동 적용, 없으면 **보강 안내**(아래 §3.1 두 처치 종류 참고) |

## 1.4 간결성·가독성 (Concision)
| 기준 | 내용 | 점검 |
|---|---|---|
| C1 분량 적정 | 항목·산출물이 과도하게 장황하지 않다(이력서는 압축 산출물 §227). | **자동(A)** — 길이 상한 대비 |
| C2 버즈워드 절제 | "열정적", "책임감 있는", "다양한 경험" 같은 막연한 버즈워드를 남발하지 않는다. | **반자동(S)** — 버즈워드 사전 매칭 |
| C3 중복 제거 | 같은 표현·내용이 여러 항목에 반복되지 않는다. | **반자동(S)** — 항목 간 n-그램/문장 유사도 |
| C4 가독성 | 한 문장이 지나치게 길거나 만연체가 아니다. | **반자동(S)** — 문장 길이 통계 |

## 1.5 일관성 (Consistency)
| 기준 | 내용 | 점검 |
|---|---|---|
| K1 어조 일관 | 항목 간 어조·문체(예: 음슴체/해요체)가 섞이지 않는다. | **반자동(S)** — 문장 종결 패턴 통계 |
| K2 표기 일관 | 같은 고유명사의 표기가 흔들리지 않는다(예: "Spring"/"스프링"). | **반자동(S)** — 변형 후보 표시 |

## 1.6 형식 충족 (Structural Completeness)
| 기준 | 내용 | 점검 |
|---|---|---|
| St1 양식 충족 | 양식의 필수 섹션 중 근거 경험이 있는데도 비어있는 항목이 없다. | **자동(A)** — 기존 양식·생성 항목 대조(§411) 재사용 |
| St2 빈 항목 없음 | 생성 성공 항목의 내용이 비어있지 않다. | **자동(A)** |

## 1.7 서식·접근성 (Presentation) — MVP 자동 개선 범위 밖
| 기준 | 내용 | 점검 |
|---|---|---|
| R1 서식·접근성 | raw URL 대신 하이퍼링크, PDF A4 규격, 표준 폰트·적정 줄간격·자간, 표·여백 정돈, 접근 가능·미만료 링크 등 시각·서식·파일 영역의 정돈. | **범위 밖** — MVP 산출물은 '텍스트 구조물'(§260)이고 활용 범위는 '열람+복사'(§347)다. 시각 디자인·PDF 내보내기는 로드맵(§504~509). 자동 개선 대상이 아니다 |

> R1 처리 방침: 폰트·줄간격·PDF 규격·하이퍼링크·링크 만료는 **서비스가 생성하지 않는 표현·파일 계층**이므로 자동 개선이 손댈 수 없다. 사용자가 복사해 외부 도구에서 서식을 입힐 때 참고하도록 **정적 '서식 체크리스트'로 고지할지**는 후속 검토 과제로 남긴다(→ 열린 질문 5). 단 raw URL이 산출물 텍스트에 그대로 등장하는 경우의 가독성 정리(예: 과도하게 긴 URL 노출 경고)는 C4 가독성 진단으로 일부 포착 가능하다.

> 범주화 근거: 신뢰성(1.1)을 최상위에 두고, 나머지는 "내용이 목표에 맞는가(1.2)·강한가(1.3)·간결한가(1.4)·일관된가(1.5)·구조가 찼는가(1.6)"로 나눈다. 1.7 서식·접근성은 **MVP 범위 밖**으로 분리해 명시한다. 자동 개선이 안전하게 손댈 수 있는 것은 주로 1.3~1.5(표현·구조·강조·간결)이며, 1.1·1.6은 **진단·고지**의 대상이지 자동으로 '채우는' 대상이 아니다(빈자리는 채우지 않고 알린다 — §469).

# 2. 나쁜 이력서 안티패턴 카탈로그

각 안티패턴을 **위반 품질 기준 → 처치 분류 → 신뢰성 가드레일 처리**로 구조화한다. **처치 분류**는 이 기능의 핵심 골격으로, 세 갈래다(도메인 용어는 §3.1에서 정의).

- **자동 적용(Auto-rewrite):** 사실(수치·고유명사·성과)을 **불변**으로 둔 채 표현·구조·강조·간결만 다듬는다. 신뢰성 검증을 통과하면 사용자가 채택할 항목 후보가 된다. (예: 약한 동사 교체, 버즈워드 제거, 수동→능동, 압축.)
- **개선 제안(Improvement Suggestion):** 자동으로 값을 만들 수 없어 **사용자 액션이 필요한 보강 안내**. 서비스는 텍스트를 바꾸지 않고 "여기에 이런 사실을 더하면 강해져요"라고 알린다. (예: 없는 수치의 객관화, 비어 있는 성과.)
- **범위 밖(Out of Scope):** 텍스트 산출물이 아니라 시각·서식·파일·외부 자산 영역이라 MVP 자동 개선이 손대지 않는다.

> 이 분류의 결정적 의미: **자동 적용은 사실을 절대 바꾸지 않는다.** "근거 있으면 구체화, 없으면 보강 안내"라는 [D] 케이스의 원칙이 바로 자동 적용과 개선 제안의 경계선이다. 같은 안티패턴이라도 *경험 기록에 근거가 있으면* 자동 적용, *없으면* 개선 제안으로 갈린다.

| # | 안티패턴(출처) | 위반 기준 | 처치 분류 | 신뢰성 가드레일 처리 |
|---|------------|---------|---------|----------------------|
| AP1 | **막연한 책임 나열** "결제 시스템을 담당했습니다" ([E]) | I1, I3 | **자동 적용** | 경험 본문에 근거 있는 행동 범위 내에서 약한 동사→강한 동사("담당"→"설계·운영"). 없는 행동·성과 부여 금지 |
| AP2 | **추상적 자기소개** "성장하는·협업 즐기는·끊임없이 고민하는" ([B]) | C2, F1 | **자동 적용**(근거 있을 때) / **개선 제안**(근거 없을 때) | 추상어를 경험·목표에 **이미 존재하는** 기술명·행동으로 구체화("협업"→"코드리뷰 주도"). 근거 경험이 없으면 구체화 대신 "이 강점을 뒷받침할 경험을 추가하세요" 제안 |
| AP3 | **모호한 규모·역할어** "중요한 역할·다양한 기술·복잡한 아키텍처 전환" ([B]) | C2 | **자동 적용** | "중요한 역할"→"적극 주도", 추상 역량→경험에 적힌 실제 행동/기술명. 표현만 구체화, 사실 불변 |
| AP4 | **버즈워드 남발** "열정적이고 책임감 있는" ([B]) | C2 | **자동 적용** | 막연한 형용사 제거·구체 표현 치환. 치환을 빌미로 없는 사실 주입 금지 |
| AP5 | **객관적이지 않은 수치화** "사용자 200% 증가·대용량 트래픽·성능 30% 개선" ([D]) | I4 | **자동 적용**(실측값 근거 있을 때) / **개선 제안**(없을 때) | 경험에 실측값(100→200명, 500tps, 300→210ms, 월 700만원)이 **있으면** as-is→to-be로 구체화. **없으면 날조 절대 금지** → "이 수치를 객관화하려면 경험에 구체 값을 적어 주세요" 제안. ★이 기능의 핵심 설계 포인트 |
| AP6 | **경험 나열형** "CI/CD 구축·EC2 관리·채팅 구현" 단순 나열 ([E]) | I1, I3 | **자동 적용** | 나열을 '왜·기여' 관점으로 재구성(기존 사실의 재배치·관점 전환). 수상은 성과란으로 분리 제안. 없는 결과 창작 금지 |
| AP7 | **정량 성과 부재** (결과가 정성으로만 서술) ([D][E]) | I3 | **자동 적용**(정성 결과 표면화) + **개선 제안**(수치 없을 때) | 경험 결과 필드에 **이미 있는** 정성 성과를 표면으로. 수치를 새로 만들지 않음. 신입은 '배운 점' 표면화 가능 |
| AP8 | **목표 무관 내용** | F2 | **자동 적용** | 작성 전략 '피할 점' 곁가지를 약화·후순위화. 항목 통째 삭제는 사용자 채택으로(자산 손실 방지) |
| AP9 | **약한 동사** "참여했다·있었다" | I1 | **자동 적용** | 행동 동사로 교체. 동사 강화로 성과 부풀리기 금지 |
| AP10 | **과도한 분량·장황함** (프로젝트마다 디테일 과다) ([A]) | C1, C4 | **자동 적용** | 군더더기 제거·문장 분할·압축(검토자는 기술역량·극복경험이 궁금). 압축 중 고유명사·수치 유실 금지 |
| AP11 | **중복** (여러 항목 같은 내용) | C3 | **자동 적용** | 한쪽으로 통합·재서술. 제거 빌미로 없는 사실 추가 금지 |

| AP13 | **수동태 남발** | I2 | **자동 적용** | 능동태 전환. 주어를 임의로 만들어 넣기 금지 |
| AP14 | **서식·접근성** raw URL·PDF 비A4·폰트/줄간격/표 여백·만료 URL ([A]) | R1 | **범위 밖** | 시각·서식·파일 계층이라 자동 개선 불가. 정적 서식 체크리스트 고지는 후속 검토(열린 질문 5). 산출물 텍스트 내 raw URL 가독성은 C4로 일부 포착 |
| AP15 | **외부 자산 의존** Github 잔디 어뷰징 ([C]) | — | **범위 밖** | 서비스가 생성하지 않는 외부 자산. 자동 개선 대상 아님(참고만) |

> 자동 개선과 신뢰성 가드레일의 경계 (핵심 원칙): **자동 적용은 "어떻게 말하는가(표현·구조·강조·간결)"만 바꾸고, "무엇이 사실인가(수치·고유명사·성과)"는 절대 바꾸지 않는다.** AP5·AP7이 보여주듯, "수치 부재"의 올바른 처치는 *수치를 만드는 것이 아니라* (a) 경험에 이미 있는 값을 자동 적용으로 구체화하거나, (b) 값이 없으면 **개선 제안으로 사용자에게 경험 보강을 안내**하는 것이다(§469 빈자리는 채우지 않고 알린다). 자동 적용과 개선 제안의 분기는 **경험 기록에 근거가 존재하는가**라는 결정적 조건으로 갈린다.

# 3. 자동 개선 기능 설계 (도메인 관점)

## 3.1 도메인 용어 정의

- **개선 기준(Quality Criteria):** §1에서 정의한 "좋은 이력서가 만족하는 조건"의 집합. 각 기준은 식별자(T1, I1 등)·범주·점검 가능성(자동/반자동/사용자검토)을 가진다. 서비스가 보유한 고정 카탈로그이며 사용자가 편집하지 않는다(MVP).
- **품질 점검(Quality Review):** 한 산출물(또는 버전)을 개선 기준에 비추어 진단하는 행위. 결과로 **품질 진단 결과(Quality Report)** 를 낳는다.
- **품질 진단 결과(Quality Report):** 한 버전에 대한 점검 결과. 위반·약점이 감지된 지점들을 **개선 소견(Finding)** 목록으로 담는다. 진단 결과는 **읽기 전용**이며 산출물을 바꾸지 않는다.
- **개선 소견(Finding):** "어느 생성 항목의, 어떤 개선 기준을, 어떻게 위반/약화했는가"를 가리키는 한 건의 지적. (예: 항목 X, 기준 I1, "약한 동사 '담당했다' 사용"). 자동 점검(A)이면 결정적 근거를, 반자동(S)이면 후보 표시를 담는다. 각 소견은 **처치 종류(자동 적용/개선 제안/범위 밖)** 를 함께 가진다.

각 개선 소견의 처치는 다음 두 종류로 나뉜다. **이 분리가 이 기능의 핵심 골격이다.**

- **자동 적용(Auto-rewrite):** 한 개선 소견에 대해 서비스가 **사실을 불변으로 둔 채** 표현·구조·강조·간결만 다듬어 만든 **항목 후보 텍스트**. 항목 단위로 생성되며, 채택 전까지는 §401의 **항목 후보(Section Candidate)** 와 동일한 격이다(버전이 아니다). 신뢰성 자동 검증과 '원본 사실 토큰 보존' 대조(§3.5·§4.4)를 통과한 것만 채택 후보로 노출된다. (예: AP1·AP3·AP4·AP6·AP8~AP13.)
- **개선 제안(Improvement Suggestion):** 자동으로 값을 만들 수 없어 **사용자 액션이 필요한 보강 안내**. 서비스는 산출물 텍스트를 바꾸지 않고, "이 경험에 구체 수치/결과를 적으면 더 강해져요" 식으로 사용자를 경험 기록 보강으로 유도한다(§469). 채택 대상이 아니라 안내 대상이다. (예: AP5·AP7의 '수치 없음' 케이스, AP2의 '근거 경험 없음' 케이스.)

> 두 종류 분기 규칙(결정): 같은 개선 소견이라도 **경험 기록에 근거가 존재하면 자동 적용, 없으면 개선 제안**으로 갈린다. 예컨대 "200% 증가"(AP5)는 경험에 실측값 "100→200명"이 있으면 자동 적용으로 객관화하고, 없으면 날조 대신 "구체 값을 적어 주세요" 개선 제안으로 돌린다. 이 분기가 신뢰성 가드레일(근거 없는 사실 금지)을 기능 골격 차원에서 강제한다.

- **품질 개선 작업(Quality Improvement Job):** 사용자가 "품질 개선"을 요청한 한 건의 비동기 작업. **자동 적용** 후보들을 생성하는 작업이며, 기존 §282 생성 작업(Generation Job)과 **동형**으로 상태(대기중·생성중·성공·실패)를 가지며 폴링으로 추적한다. 개선 제안은 외부 LLM 호출 없이 진단 단계에서 산출되므로 이 작업을 거치지 않는다.

> 명명 근거: 기존 모델의 **생성 항목·항목 후보·버전·생성 작업** 어휘를 그대로 재사용해, 자동 적용을 "새로운 종류의 항목 재생성"으로 환원한다. 이로써 버전·채택·가드레일 모델을 새로 만들지 않고 §308·§389·§277에 그대로 얹힌다. 개선 제안은 텍스트를 바꾸지 않는 안내이므로 버전·가드레일과 무관하다.

## 3.2 개선 대상 (결정)

**MVP 자동 개선 대상은 이력서로 한정하고, 포트폴리오는 로드맵으로 미룬다.**

| 트레이드오프 | 이력서만 | 이력서+포트폴리오 |
|---|---|---|
| 기준 적합성 | 이력서는 압축·정형(§227)이라 간결성·강한 동사·버즈워드 기준이 정확히 들어맞음 | 포트폴리오는 서사·심화(§257)라 "간결성·압축"이 오히려 가치를 깎을 수 있어 기준이 부분적으로만 적용 |
| 구현 비용 | 양식·섹션 모델 위에 바로 얹힘 | 포트폴리오용 별도 기준 가중치 필요 |
| 사용자 가치 | "어디 고칠지 모름"(P4)이 이력서에서 가장 첨예 | 추가 가치는 있으나 기준 재정의 선행 |

> 결정 근거: 이 문서가 정의한 개선 기준의 다수(간결성·강한 동사·버즈워드·일관성)는 **압축 산출물인 이력서**에 정밀하게 맞는다. 포트폴리오는 "장황함=나쁨"이 성립하지 않아 같은 기준을 그대로 쓸 수 없다. MVP는 이력서에 집중하고, 포트폴리오 자동 개선은 서사용 기준(논리 흐름·근거 충실도 등)을 별도 정의한 뒤 더한다(로드맵). 단, **신뢰성 검증(T1·T2)은 포트폴리오 생성에도 이미 적용**되므로 그 부분은 변화 없다.

## 3.3 개선 단위 (결정)

**개선은 산출물 전체를 점검하되, 처치(다듬기)는 생성 항목 단위로 한다.**

- 품질 점검은 산출물(활성 버전)을 통째로 본다 — 중복(C3)·일관성(K1·K2)은 항목을 가로질러야 보이기 때문이다.
- 처치는 항목 단위 후보 생성으로 한다 — 기존 §396 "한 항목 변경이 다른 항목을 바꾸지 않는다"와 §401 항목 후보·채택 모델에 그대로 맞는다.

> 근거: 진단의 시야(산출물 전체)와 처치의 입도(항목)를 분리하면, 항목 간 기준(중복·일관성)을 진단하면서도 채택·버전 모델은 기존 항목 단위 구조를 깨지 않는다.

## 3.4 트리거 방식 (결정 — 온디맨드 단일 채택, 1차 생성 자동 1패스는 로드맵)

세 후보를 비교한다.

| 방식 | 비용 가드레일 정합 | 신뢰성 | UX | 비동기 잡 모델 정합 |
|---|---|---|---|---|
| **(가) 1차 생성 직후 자동 1패스** | 1차 생성마다 개선 호출이 강제로 따라붙어 **비용이 사실상 배증**. 차감 모델도 모호해짐 | 자동 개선도 검증을 타므로 안전하나, 사용자가 원본 초안을 보기 전에 손이 가 **"내 글이 멋대로 바뀐다"는 통제감 상실** | 사용자가 누른 적 없는 변경이 생김(다크패턴 인접 — UX 핵심 §4 죄책감·통제권) | 1차 생성 잡에 개선 잡이 묶여 잡 모델이 복잡 |
| **(나) 온디맨드 "품질 개선" 버튼** | 사용자가 명시적으로 누를 때만 호출 → 비용 통제 명확, 별도 상한 부여 가능 | 사용자가 원본을 본 뒤 스스로 선택 → 통제감 유지 | "어디가 약한지" 진단을 먼저 보고 개선을 결정 → P4를 정확히 해소, One Thing per Page와 정합 | 기존 생성 잡과 동형의 별도 잡으로 깔끔히 추가 |
| **(다) 둘 다** | (가)의 비용 문제를 그대로 안음 | — | 선택지 증가로 인지 부하(UX 핵심 §1) | 구현 복잡도 최대 |

**결정: (나) 온디맨드 "품질 개선"을 MVP로 채택한다.** 단, **점검(진단)과 개선(처치)을 2단계로 분리**한다.

1. 사용자가 산출물 열람 화면에서 "품질 점검"을 누르면 → **품질 진단 결과**(개선 소견 목록)를 보여준다. 자동(A) 점검은 즉시·무비용에 가깝게, 반자동(S)·LLM 보조가 필요한 점검만 비동기 처리.
2. 사용자가 소견을 보고 "이대로 다듬기(품질 개선)"를 누르면 → **품질 개선 작업**으로 항목 후보들을 생성하고, 사용자가 항목별/일괄로 채택한다.

> 결정 근거: (가)는 비용 배증과 통제감 상실이라는 두 치명적 비용을 지닌다. UX 핵심 가이드 §4(죄책감·다크패턴 금지)와 §1(인지 부하)에 비추어, "사용자가 보기 전에 멋대로 바꾸는" 자동 1패스는 MVP에 부적합하다. (나)는 P4("어디를 고칠지 모름")를 **진단 먼저, 처치 나중**의 2단계로 가장 정확히 해소하고, 기존 비동기 잡·항목 후보·채택 모델에 그대로 얹힌다. 1차 생성 직후 자동 1패스는 비용·통제권 문제를 해결할 운영 정책(예: 옵트인, 무료 1회)이 정해진 뒤 로드맵으로 더한다. → **열린 질문 1**.

## 3.5 버전·가드레일 정합 (결정)

- **개선 결과는 새 버전이 된다.** 항목 후보를 채택하면 §396대로 직전 활성 버전을 복제한 위에 채택 항목만 교체한 새 버전이 만들어지고 활성이 된다. 자동 개선은 "또 다른 경로의 항목 재생성"이므로 별도 버전 차원을 만들지 않는다. 일괄 채택(여러 항목 동시 개선 채택)은 **한 번의 버전 전이**로 묶어 버전 폭증을 막는다.
- **비용 가드레일 차감:** 품질 개선은 외부 LLM 호출이므로 비용이 발생한다. **품질 개선 작업이 성공해 채택 가능한 후보가 하나라도 나오면 1회 차감**한다(전 항목 실패 시 미차감 — §448 불변식 동형). 단, 항목 재생성 상한과 **별도 상한**을 둘지, 1차 생성/재생성 상한에 합산할지는 운영 수치 결정이다 → **열린 질문 2**. **품질 점검(진단)만 하고 개선을 누르지 않으면**, 자동(A) 점검은 무비용이므로 미차감, 반자동(S)에 LLM 보조를 쓰는 점검은 차감 여부를 운영 정책으로 분리한다(권장: 진단은 경량·무차감, 처치만 차감).
- **신뢰성 자동검증을 개선 결과에 반드시 적용한다.** 개선으로 만든 항목 후보도 §473 자동 검증(수치·고유명사 문자열 대조)을 동일하게 통과해야 채택 후보가 된다. 검증 실패 시 §481대로 자동 1회 재생성 → 재실패 시 그 항목은 개선 제안에서 제외하고 사용자에게 알린다(원본 항목 유지).

> 신뢰성 검증 적용 근거(반드시): 자동 개선의 처치는 LLM이 수행하므로, "다듬기"라는 의도에도 불구하고 수치·고유명사를 새로 끼워넣거나 변형할 위험이 구조적으로 존재한다. 기존 생성·재생성과 **동일한 결정적 검증**을 개선 결과에 적용하지 않으면, 이 기능이 오히려 신뢰성 가드레일에 구멍을 낸다. 따라서 개선 결과 검증은 선택이 아니라 **필수**다. 더 나아가 개선 후보는 **원본 항목의 사실 토큰 집합(수치·고유명사)을 보존**하는지도 대조해, "다듬다가 사실을 흘린"(누락) 경우도 검증실패로 본다(개선의 특수 불변식).

## 3.6 사용자 시나리오

- **성공(진단→개선):** 사용자가 완성된 이력서를 열람하다 "품질 점검"을 누른다. 잠시 후(자동 점검 즉시 + 비동기 점검 폴링) "강한 동사로 바꾸면 좋아요(3곳)", "버즈워드가 보여요(2곳)", "두 항목 내용이 겹쳐요" 같은 소견이 친근한 언어로 뜬다. "이대로 다듬기"를 누르면 항목별 개선 후보가 만들어지고, 원본과 나란히 비교해 마음에 드는 것만 채택한다. 채택분이 새 버전이 된다.
- **성공(부분 채택):** 다섯 소견 중 셋만 채택한다. 나머지 항목은 그대로 유지된다(§334 한 항목 재생성이 다른 항목을 바꾸지 않는다).
- **성공(개선 제안):** 소견에 "이 경험에 결과·수치를 적으면 더 강해져요"가 뜬다(AP5·AP7). 이는 **자동 적용이 아니라 개선 제안**이다 — 서비스가 수치를 만들지 않고 사용자를 경험 기록 보강으로 안내한다(§469). 사용자가 경험에 실측값을 채워 재생성하면 그때는 자동 적용으로 객관화된다.
- **실패(개선 결과 검증 실패):** 개선 후보가 신뢰성 검증을 통과하지 못하면(예: 다듬다 고유명사 표기를 바꿈) 자동 1회 재시도 후에도 실패하면 그 항목은 제안에서 빠지고 "이 부분은 안전하게 다듬기 어려워 원본을 유지했어요"라고 알린다(§481 회복 정책 동형, 부정 감정 최소화).
- **실패(개선할 게 없음):** 점검 결과 소견이 0건이면 "지금도 충분히 좋아요. 고칠 만한 곳을 못 찾았어요" 식의 긍정 빈 상태를 보여준다(UX 핵심 §9 행동 유도형 빈 화면, 죄책감 없이).
- **실패(개선 한도 초과):** 품질 개선 호출이 상한에 도달하면 §330·§451대로 남은 횟수·회복 시점·대안(직접 편집)을 안내한다.

## 3.7 측정 가능한 수용 기준

- QC1. 사용자가 이력서 산출물에 품질 점검을 요청하면, §1의 자동(A) 기준 위반은 결정적으로 진단되어 개선 소견으로 제시된다(예: 길이 상한 초과, 빈 필수 항목).
- QC2. 같은 산출물이라도 약한 동사·버즈워드·수동태·중복이 더 많은 쪽이 더 많은 개선 소견을 받는다(사람이 비교해 식별 가능).
- QC3. 품질 개선으로 만든 항목 후보는 §473 자동 신뢰성 검증을 통과한 것만 채택 후보로 노출된다. 검증 실패 후보는 노출되지 않고, 원본 항목이 유지된다.
- QC4. 품질 개선 후보는 원본 항목의 수치·고유명사 사실 토큰을 누락·변형하지 않는다(추가 0건 + 보존 — 결정적 대조).
- QC5. 항목 후보 채택은 직전 버전을 보존한 채 새 활성 버전을 만들고, 채택하지 않은 항목은 변하지 않는다(§334·§396).
- QC6. 품질 개선 작업은 비동기로 접수되어 폴링으로 추적되며, 성공 시에만 채택 가능한 후보가 연결된다(§299 동형). 진행 중 작업이 영원히 멈추지 않는다(시간 초과 시 실패 수렴).
- QC7. 개선 결과(채택 가능한 후보 ≥ 1)가 나오면 비용 가드레일이 1회 차감되고, 전 항목 실패 시 미차감된다.
- QC8. 타인의 산출물은 점검·개선할 수 없다(소유 격리 — §302·§383).
- QC9. 자동 적용은 어떤 경로로도 경험 근거 밖의 새 수치·고유명사·성과를 추가하지 않는다(§534 기준 16·17과 동일 불변식).
- QC10. 포트폴리오 산출물에는 (MVP에서) 품질 개선 진입점이 노출되지 않는다.
- QC11. 모호 수치·규모어(AP5)가 검출될 때, 경험 기록에 대응 실측값이 **있으면 자동 적용 후보**로, **없으면 개선 제안(보강 안내)** 으로 분기되며, 후자는 산출물 텍스트를 바꾸지 않는다(자동 적용 vs 개선 제안 분기의 결정적 동작).
- QC12. 서식·접근성(R1, AP14)·외부 자산(AP15)은 자동 적용·개선 제안 어느 후보도 만들지 않는다(MVP 범위 밖 — 진단·고지로만 다룬다).

## 3.8 기존 파이프라인 통합 지점 (결정 — 별도 개선 패스 호출)

`ClaudeCliArtifactGenerationAdapter`를 읽고 현실적으로 판단한다.

두 통합 방식을 비교한다.

| 방식 | 내용 | 트레이드오프 |
|---|---|---|
| **(가) 프롬프트 보강** | 1차 생성 프롬프트에 "강한 동사·간결성·버즈워드 절제" 지시를 더 넣어 처음부터 더 좋게 생성 | 비용 무증가·간단. 단 **사후 진단·선택적 처치 불가**(P4의 "어디가 약한지 보여주고 고르게" 가치 상실). 이미 만들어진 산출물엔 소급 불가 |
| **(나) 별도 개선 패스 호출** | 점검은 결정적 검사+경량 보조로, 처치는 **원본 항목 텍스트를 입력으로 받는 별도 LLM 호출**로 항목 후보 생성 | 비용 추가·구현 추가. 단 진단→선택→처치의 2단계와 사후 적용을 모두 지원 |

**결정: (나) 별도 개선 패스를 핵심으로 채택하되, (가)를 보완으로 병행한다.**

- 처치는 **새 포트(예: `QualityImprovementPort`) + 새 어댑터**로 분리한다. 입력은 *경험 묶음이 아니라* **개선 대상 항목의 원본 텍스트 + 작성 전략 + 적용할 개선 소견(어떤 기준을)** 이고, 출력은 다듬어진 후보 텍스트 + (검증용) 사실 토큰이다. 프롬프트의 **신뢰성 절대 규칙·factGroundings 산출 구조는 기존 어댑터(45~52행)에서 그대로 가져와 재사용**한다 — "다듬되 없는 사실 금지, 근거 함께 산출"이 동일하게 필요하기 때문이다.
- 동시에 1차 생성 프롬프트((가))에도 강한 동사·간결성 지시를 가볍게 보강해, **처음부터 개선 소견이 적게 나오도록** 한다(점검·처치 부하 절감). 이는 기존 `appendResumeInstructions`(96~109행)에 지시문을 더하는 저비용 변경이다.

> 현실 근거: 기존 어댑터는 "경험 묶음 → 생성"의 단방향이라 사후 진단·선택적 처치가 없다. P4를 정확히 닫으려면 "이미 만든 산출물을 진단하고 부분 처치"가 필수이므로, 경험 입력이 아닌 **항목 텍스트 입력**의 별도 패스가 본질이다. 다만 그 패스도 신뢰성 규칙을 그대로 짊어져야 하므로(§3.5), 기존 프롬프트의 가드레일·근거 산출 부분을 공유 컴포넌트로 빼 재사용하는 것이 자연스럽다.

# 4. 구현 영향 범위 요약 (개발팀장·개발자용)

이 절은 결정이 아니라 **영향받는 곳의 지도**다. 구체 설계·테스트 작성은 후속 단계에서 한다.

## 4.1 백엔드
- **도메인 모델(`artifact`/신규 `quality` 패키지):**
  - 개선 기준(Quality Criteria) 카탈로그 — 코드 상수/enum + 점검기(checker) 모음. 각 기준은 점검 가능성(A/S/U)을 메타로 가짐.
  - 품질 진단 결과(Quality Report)·개선 소견(Finding) VO — 읽기 전용, 항목 식별자·기준·근거 텍스트를 담고, **처치 종류(자동 적용/개선 제안/범위 밖)** 를 필드로 가짐. 처치 종류 분기(근거 경험 존재 여부)는 진단 단계에서 결정한다.
  - 품질 개선 작업(Quality Improvement Job) — 기존 `GenerationJob`과 동형 엔티티(상태·소유자·산출물 연결·시간초과 수렴). **자동 적용** 소견만 입력으로 받아 항목 후보를 생성한다(개선 제안은 이 작업을 거치지 않는 진단 산출물).
- **포트·어댑터:** 신규 `QualityImprovementPort` + Claude CLI 어댑터. 기존 `ClaudeCliArtifactGenerationAdapter`의 신뢰성 규칙 블록(45~52행)·OUTPUT_SCHEMA의 factGroundings 구조를 공유 컴포넌트로 추출해 재사용.
- **검증:** 기존 자동 신뢰성 검증(수치·고유명사 추출·대조)을 개선 결과에도 호출. **추가로 "원본 사실 토큰 보존" 대조 신설**(개선 특유 불변식 — QC4).
- **서비스(유스케이스):** 품질 점검(진단, 트랜잭션 내 결정적 검사 + 트랜잭션 밖 경량 LLM 보조), 품질 개선 작업 접수·워커(기존 §277 비동기 워커 패턴 재사용, 트랜잭션 분리 가이드대로 LLM 호출은 트랜잭션 밖), 후보 채택(기존 채택 유스케이스 재사용·일괄 채택 추가).
- **가드레일:** 품질 개선 차감 카운터(별도 상한 vs 합산은 운영 결정).

## 4.2 프론트
- **화면:** 산출물 열람 화면에 "품질 점검" 진입점, 진단 결과(개선 소견 목록) 패널, 원본↔개선 후보 비교 뷰, 항목별/일괄 채택 컨트롤. (UX 핵심: One Thing per Page — 진단과 처치를 한 화면에 욱여넣지 않고 단계 분리.)
- **상태·폴링:** 품질 개선 작업 폴링(기존 생성 작업 폴링과 동형), 진단 로딩은 스켈레톤(UX 핵심 §7).
- **두 처치 종류의 시각 구분:** **자동 적용** 소견은 "이대로 다듬기"(원본↔후보 비교·채택)로, **개선 제안** 소견은 채택 버튼이 아니라 "경험 보강하러 가기"(해당 경험 기록으로 이동)로 다르게 노출한다. 둘을 섞으면 사용자가 "왜 어떤 건 자동으로 안 바뀌나"를 혼동한다(UX 핵심 §1·§8 예측 가능한 CTA).
- **빈 상태·에러:** "고칠 곳 없음" 긍정 빈 상태, 검증 실패 항목 "원본 유지" 안내, 한도 초과 안내(에러 가이드: 대안 제시·부정 감정 최소화).

## 4.3 마이그레이션
- 품질 개선 작업 영속 테이블(상태·소유자·산출물·시각) → **Flyway V5+ 신규 마이그레이션 필요**(메모리: 스키마 변경은 ddl-auto 아닌 V2+ 마이그레이션으로만, 운영 ddl-auto=validate). 진단 결과를 영속할지(매번 재계산 vs 저장)는 설계 결정 — 권장: 진단은 휘발(요청 시 계산), 작업·후보만 영속.
- 항목 후보·버전은 기존 테이블 재사용(개선을 항목 재생성으로 환원했으므로 신규 산출물 스키마 변경 최소).

## 4.4 테스트
- **결정적 검증 테스트(핵심):** 개선 후보가 원본 사실 토큰을 보존하는가(QC4), 새 수치·고유명사 추가 0건인가(QC9) — LLM 없이 시밍(seam)으로 검증(메모리: 테스트 비용 0, 포트 시밍).
- **잡 동형성 테스트:** 비동기 접수·폴링·시간초과 수렴(QC6), 소유 격리(QC8).
- **버전 정합 테스트:** 채택 후 새 활성 버전·미채택 항목 불변(QC5).
- **가드레일 테스트:** 채택 가능 후보 ≥1 시 차감, 전 실패 시 미차감(QC7).
- **E2E:** 진단→개선→채택 핵심 흐름(메모리 교훈: 전 흐름 자동 E2E 부재가 산출물 미저장 버그를 놓쳤음 — 개선도 채택·영속까지 E2E로 닫는다).

# 5. 오너 결정 사항

## 5.1 확정 (2026-06-23 오너 승인 — 구현은 이 결정을 따른다)
1. **트리거 방식:** §3.4의 **온디맨드 2단계**(품질 점검→소견 확인→다듬기 채택)를 MVP로 확정. 1차 생성 직후 자동 1패스는 미도입(로드맵).
2. **개선 대상:** **이력서 한정**으로 확정. 포트폴리오 자동 개선은 로드맵(신뢰성 검증 T1·T2는 이미 적용되어 변화 없음). 포트폴리오 산출물엔 품질 개선 진입점을 노출하지 않는다(QC10).
3. **비용 정책:** **진단(품질 점검)은 무차감**, **처치(품질 개선)는 항목 재생성 상한과 별개의 자체 일일 한도에서 채택 가능 후보 ≥1일 때 1회 차감**(전 항목 실패 시 미차감 — QC7). 구체 N회 수치는 운영 정책(아래 5.2-1).

## 5.2 미해결 (오너 후속 결정 — MVP 구현은 시드/플레이스홀더로 진행)
1. **품질 개선 일일 상한 수치:** 5.1-3에서 '별도 상한 존재'는 확정. 구체 N회는 운영·예산 판단(§549 동류). MVP는 설정값(프로퍼티)으로 외부화하고 시드값으로 진행.
2. **개선 기준 사전·임계값 큐레이션:** 약한 동사·버즈워드·피할 점 사전, 길이 상한 수치, 문장 길이·유사도 임계값 등 반자동(S) 기준의 구체 사전·임계값. MVP는 §553 프리셋 시드처럼 **잠정 시드 사전/임계값**을 두고(설정 외부화 권장), 오너 큐레이션 시 교체.
3. **포트폴리오 자동 개선 시점·기준:** 서사용 개선 기준(논리 흐름·근거 충실도)을 언제 정의해 도입할지(로드맵).
4. **서식·접근성(R1·AP14) 정적 체크리스트 고지 여부:** MVP는 텍스트 산출물이라 서식(폰트·줄간격·PDF A4·하이퍼링크·링크 만료)을 자동 개선하지 못한다. 사용자가 복사해 외부에서 서식을 입힐 때 참고할 정적 체크리스트를 결과물 활용 화면(§347)에 고지할지, 시각 디자인·PDF 로드맵까지 미룰지.
