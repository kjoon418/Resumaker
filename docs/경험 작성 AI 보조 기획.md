# 경험 작성 AI 보조 기획

> **상태:** 1차 슬라이스(결정적 경험 점검) 구현 완료 · AI 인터뷰형 보조는 후속
> **관련 도메인:** 기능 1 경험 기록, 핵심 목적 2(백지의 공포 최소화)
> **관련 파일:** `docs/Resumaker 도메인 이해.md` §1 경험 기록, §MVP 경계와 로드맵

---

## 구현된 1차 슬라이스 — 결정적 경험 점검 (무LLM)

"AI 인터뷰형 보강"의 **결정적(무LLM) 첫 슬라이스**를 먼저 구현했다. AI가 사실을 지어낼 위험이 0이면서, 이력서가
자동으로 못 고치는 문제의 **선행 조건**을 즉시 짚어 준다.

### 왜 결정적 점검부터인가
이력서 품질 점검의 **'모호한 수치'(VAGUE_METRIC)** 는 출처 경험에 구체 수치가 없으면 자동 개선이 **불가능**하다.
이력서 점검은 이 경우 "경험 보강하러 가기"로 사용자를 경험 편집 화면으로 보내지만, 정작 **경험 자체는 진단되지 않아**
무엇을 고쳐야 할지 안내가 없었다. → 경험을 같은 결정적 검사기로 점검해 루프를 닫는다.

### 동작 (서버 `experience.application.ExperienceReviewService`)
경험은 '사실의 원천'이라 신뢰성 가드레일상 자동 재작성을 하지 않는다. 점검은 전부 **보강 유도(elicitation)** —
무엇을 더 적으면 좋은지 알려줄 뿐 값을 짓지 않는다. 무LLM·무차감·결정적(같은 입력 → 같은 결과).

코퍼스 = 제목 + 본문 + STAR + 역량 태그(이력서 `QualityReviewService.corpusOf`와 동형). MVP 점검 3종:

| 기준 | 조건 | 보강 유도 |
|------|------|----------|
| `VAGUE_METRIC` | 규모어 사전 매칭 **그리고** 구체 수치 전무 | "구체 수치를 적으면 이력서에서 더 강해져요(예: 대용량→초당 500건)" |
| `MISSING_RESULT` | 결과(result) 칸이 비어 있음 | "이 경험으로 무엇이 달라졌는지(성과)를 적어보세요" |
| `THIN_BODY` | 본문이 임계(트림 20자) 미만 | "내용이 짧아요. 문제를 어떻게 해결했는지 한 줄 더 적어보세요" |

> `VAGUE_METRIC`은 수치가 **하나라도 있으면 억제**한다 — 이력서가 자동 객관화할 수 있으므로(이력서 점검의
> `hasNumericEvidence`와 동형). 규모어 사전·수치 추출기(`QualityCriteriaDictionary`·`FactTokenExtractor`)를
> 이력서 점검과 **그대로 공유**해 기준을 단일 원천으로 둔다.

### 노출
- **경험 편집 화면 인라인 점검 패널:** 저장된 경험을 열 때 `GET /experiences/{id}/review`로 소견을 받아 친근하게
  안내한다. 소견 0건이면 비난/공허 대신 긍정 빈 상태("지금도 좋아요"). 신규(미저장) 경험엔 점검을 숨긴다.
- **경험 목록 '보강 추천' 배지:** 목록 응답 `ExperienceResponse.boostHintCount`(점검 소견 수)가 0보다 크면 배지를
  단다. 목록 조회 한 번에 함께 실려 추가 호출이 없다.
- **역링크:** 이력서의 "경험 보강하러 가기"가 이미 편집 화면으로 이동하므로, 패널이 그 빈틈을 자동으로 짚는다.

### 범위 밖 (아래 AI 인터뷰형 보조가 후속 레이어)
LLM 기반 개인화 질문·표현 초안, 점검 결과 영속, 약한 동사·버즈워드 등 표현 점검 확장.

## 문제 정의

### 현재 상황
MVP에서 경험 기록의 "백지의 공포 최소화"는 **정적 회상 보조**(예시 플레이스홀더 + 유도 질문 세트)로 실현된다. 이는 AI 호출 없이 즉시 제공 가능하며, 현재 구현 완료된 상태다.

### 잔여 페인포인트
정적 회상 보조만으로 해소되지 않는 상황이 존재한다.

| 상황 | 사용자가 겪는 어려움 |
|------|-------------------|
| 경험은 있는데 어떻게 표현해야 할지 모르겠다 | "뭔가 했는데 말이 안 나와요" — 질문을 읽어도 백지에서 첫 문장을 시작하기 어렵다 |
| 두루뭉술하게 적어 놓고 충분한지 모르겠다 | "이 정도면 됐나요?" — 자신의 기록이 생성에 충분한 재료가 되는지 확신하지 못한다 |
| 나중에 보강하려 했는데 무엇을 더 쓸지 모른다 | STAR 구조 선택 항목을 채우려는 의지는 있으나 어디서 막히는지 스스로 모른다 |

이 공백은 도메인 이해 §1의 로드맵 항목인 **"AI 인터뷰형 보강"** 으로 식별되어 있다.

## 목표 UX

사용자가 경험의 본문이나 STAR 항목을 작성하는 도중 또는 작성 후에, AI가 다음 두 가지를 제공한다.

1. **초안 보완 제안:** 사용자가 적은 내용을 바탕으로 "이렇게 표현하면 어떨까요?" 형태의 문장 초안을 제시한다. 사용자가 수용·거부·수정할 수 있으며 강제하지 않는다.
2. **추가 질문 제시:** 현재 기록에서 빠진 맥락(성과 수치, 구체 행동, 기간 등)을 파악해 "이 부분도 기억나시면 추가해 주세요." 식의 개인화된 유도 질문을 1~3개 던진다. 정적 유도 질문보다 구체적이고 해당 경험에 맞춤화된다.

**핵심 원칙:**
- AI 제안은 **표현 보조**에 한정한다. 사용자가 적지 않은 사실(수치, 성과)을 AI가 생성해 삽입하지 않는다(신뢰성 가드레일 §2 준수).
- 사용자가 제안을 무시하고 직접 작성해도 어떤 차단도 없다.
- 제안 생성 중에도 저장 가능하다(비동기 경험 보조, 저장 흐름을 막지 않음).

## 기존 생성 파이프라인 재사용 방안

현재 이력서·포트폴리오 생성은 `ClaudeCliClient`(`server/.../generation/infrastructure/ClaudeCliClient.kt`)를 통해 Claude CLI를 셸 호출한다. 경험 작성 AI 보조도 동일한 파이프라인을 재사용할 수 있다.

| 구성 요소 | 재사용 방안 |
|----------|-----------|
| `ClaudeCliClient.complete(prompt, jsonSchema)` | 경험 본문·STAR 항목을 입력으로, 초안 제안·추가 질문을 JSON으로 받는 새 어댑터 작성 |
| `ClaudeCliException` → `GlobalExceptionHandler` 503 매핑 | 이미 구현 완료. 보조 기능 실패 시 동일하게 503으로 노출, 클라이언트가 "지금은 AI 보조를 사용할 수 없어요" 처리 |
| 비용 가드레일 | 경험 보조 호출도 카운트 대상에 포함할지 여부를 결정해야 한다(열린 질문 참고) |
| `ProcessRunner` seam | 테스트 시 실제 CLI를 호출하지 않는 구조 그대로 적용 가능 |

새로 필요한 것:
- 경험 보조 전용 프롬프트 템플릿 (신뢰성 가드레일 지시 포함)
- 경험 보조 응답 JSON 스키마 (제안 문장 목록 + 추가 질문 목록)
- 경험 보조 어댑터 / 서비스 계층
- 프론트에서 비동기로 제안을 요청·표시하는 UX

## 단계적 범위

### MVP+1 (최소 구현, 가치 검증 목적)
- 경험 본문을 다 적은 뒤 "AI 보완 제안 받기" 버튼을 누르면 초안 제안 1~2문장과 추가 질문 1~3개를 표시한다.
- 제안 수용 시 본문 칸에 삽입(커서 위치), 거부 시 그냥 닫힌다.
- 실패(503) 시 조용한 인라인 안내("지금은 AI 보조를 사용할 수 없어요."), 기존 작성 내용 유지.

### 확장 (가치 검증 후)
- 사용자가 입력을 멈출 때 자동 트리거(디바운스), 명시적 버튼 제거.
- STAR 항목별 개인화 질문(예: "결과 항목이 비었어요. 어떤 변화가 있었나요?").
- 제안 이력 보존(이번 보조 결과가 경험 기록과 함께 저장되어 다음 편집 시 참고 가능).
- 비용 가드레일 연동(경험 보조 호출 횟수 상한 및 사용자 안내).

## 도메인 영향

### 기존 도메인 규칙과의 정합
- **신뢰성 가드레일:** AI 보조가 생성한 표현 제안은 산출물이 아니라 사용자 기록의 편집 보조다. 최종 경험 기록에 들어가는 내용은 여전히 사용자가 확정한다. 직접 편집에 자동 검증을 적용하지 않는 기존 규칙(도메인 이해 §신뢰성 가드레일)이 여기에도 동일하게 적용된다.
- **비용 가드레일:** 경험 보조 호출이 1차 생성·항목 재생성 카운터와 별도로 집계될지, 통합 상한에 포함될지는 도메인 규칙 결정이 필요하다(열린 질문 #1 참고).
- **경험 기록 도메인 모델:** 경험 기록 엔티티에 AI 보조 이력 필드를 추가하는지, 또는 별도 엔티티로 분리하는지는 구현 설계 단계에서 결정한다. 기획 단계에서는 경험 기록과 보조 결과가 연결된다는 것만 확정한다.

### 기존 코드에 미치는 영향
- `GlobalExceptionHandler`: 변경 없음(ClaudeCliException 503 매핑이 이미 경험 보조에도 적용됨).
- `ClaudeCliClient`: 변경 없음(새 어댑터가 기존 `complete()` 호출).
- 경험 기록 도메인/엔티티: 확장 가능성 있음(보조 이력 연결 여부에 따라).
- 프론트(`app/webApp`): 경험 편집 화면에 보조 UX 추가 필요.

## 열린 질문

1. **비용 가드레일 범위:** 경험 작성 AI 보조 호출을 기존 1차 생성·재생성 카운터에 합산할지, 별도 상한(예: 하루 경험당 N회)으로 분리할지. 사용 빈도와 단가가 생성보다 낮을 수 있어 별도 관리가 유리할 수 있다.
2. **트리거 방식:** 명시적 버튼(사용자가 직접 요청) vs. 자동 디바운스(입력 멈춤 후 자동 제안). 명시적 버튼이 의도 확인·비용 예측에 유리하나, 자동 트리거가 UX 흐름이 자연스럽다.
3. **제안 수용 방식의 범위:** 제안을 기존 본문에 "추가"할지, "교체"할지, "참고용으로만 노출"할지. 교체는 사용자 기록 손실 위험이 있어 별도 확인이 필요하다.
4. **경험 보조 결과의 저장 여부:** 보조가 제시한 제안·질문을 경험 기록과 함께 저장해 다음 편집 세션에서도 볼 수 있게 할지, 세션 내 임시 표시에 그칠지. 저장 시 도메인 모델 확장이 필요하다.
5. **포트폴리오 특화 보조:** 포트폴리오는 서사·심화가 핵심이므로, 이력서용 압축 제안보다 "상황→행동→결과 서사를 더 풀어보세요" 식의 심화 질문이 더 적합할 수 있다. 이력서·포트폴리오 생성 목적에 따라 보조 프롬프트를 분기할지 여부.
