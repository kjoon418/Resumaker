이 문서는 Spring Boot 애플리케이션을 구현할 때, 검증 로직을 어떻게 배치해야 하는지를 명시하기 위해 작성되었다.

# 목표
1. 동일한 검증 로직이 여러 곳에서 중복되지 않도록 한다. (ex_ 도메인이 검증하는 내용을 컨트롤러에서도 검증하지 않도록 한다)
2. 검증 로직의 관리 포인트를 가능한 한 줄인다. (ex_ 도메인 생성시 항상 적용될 규칙이라면, 도메인 생성자에서만 검증하도록 한다)
3. 한 부분의 검증 로직이 과하게 복잡해지지 않도록 한다. (ex_ 도메인이 복잡해질 수 있으니, 필수 값의 null 체크는 DTO에 위임한다)

# 검증 로직의 분리

## Controller
Controller는 API의 계약을 표현한다. 즉, VO나 도메인에서 `null`을 체크하고 있더라도, 필수 파라미터에 대한 `null`체크를 반드시 수행한다.

여기서 검증할 내용:
1. 필수 값이 누락 여부. 즉, `null`일 수 없는 값이 `null`이 아님을 검증한다.

여기서 검증하지 않을 내용:
1. 세부적인 비즈니스 요구사항. (ex_ "회원의 이름은 5자 이내여야 한다.")

검증 방식:
1. DTO의 경우, Bean Validation을 이용해 검증한다.

## Service DTO
여기서 Service DTO는, `...Command`처럼 서비스 계층이 웹 계층에 의존하지 않도록, 서비스 메서드의 파라미터에서 사용되는 DTO를 의미한다.
(또한, 컨트롤러 계층에서 받은 `String` 값이 서비스에 그대로 넘어오지 않도록, VO로 값을 받는다)

Service DTO는, 서비스가 요청을 수행하기 위해 어떤 값을 받아야 하는지를 정의한다.
(즉, 원시 값 대신 VO를 전달하도록 강제한다)
`null`일 수 없는 값을 `null`로 보내는 것은 계약 위반으로 보고, 별도로 검증하진 않는다.

여기서 검증하지 않을 내용:
1. 필수 값의 누락 여부 (Controller DTO에서 검증하므로, 중복 검증이 된다)

## 순수 도메인, 엔티티, VO

서비스는 이 분류에 해당하지 않는다.

여기서 검증할 내용:
1. 애플리케이션 전역에 항상 적용될 요구사항 (ex_ "회원의 이름은 비어 있을 수 없다.")
2. 해당 도메인을 사용할 때 항상 적용될 요구사항 (ex_ "회원을 생성할 때 항상 적용할 규칙", "회원을 수정할 때 항상 적용할 규칙")

여기서 검증하지 않을 내용:
1. 반드시 데이터베이스 등 외부 인프라에 의존해야 검증할 수 있는 요구사항 (ex_ "회원 이름은 중복될 수 없다.")

검증 방식:
1. 객체의 불변식은 생성자나 정적 팩터리 메서드에서 검증한다.
    1. 인스턴스 생성에 항상 적용될 규칙은 생성자에서 검증한다.
    2. 생성 맥락에 따라 다르게 적용될 규칙은 정적 팩터리 메서드에서 검증한다.
2. 검증 로직은, 그 목적에 따라 하나의 `private` 메서드로 응집한다. (ex_ `validateUpdatable`)
    - 단, 검증 로직이 간결해서 메서드 분리가 오히려 가독성을 떨어트릴 수 있다면, 굳이 분리하지 않는다.

"객체의 불변식은 생성자나 정적 팩터리 메서드에서 검증한다."의 구체적인 예시는 다음과 같다.

```java
public class User {

    private User(/* ... */) {
        // 회원 인스턴스 생성 시 항상 적용될 규칙
    }

    public static User create(/* ... */) {
        // 새로운 회원을 만들 때만 적용될 규칙
    }

    public static User retrieve(/* ... */) {
        // 기존 회원 정보로 인스턴스를 만들 때 적용될 규칙
    }
}
```

## 서비스

여기서 검증할 내용:
1. 외부 인프라(데이터베이스 등)에 의존해야 검증할 수 있는 요구사항
2. 도메인에 위임할 경우, 강결합되지 않았던 도메인이 강결합되는 요구사항

검증 방식:
1. `validate...`이라는 네이밍의 `private` 메서드를 호출해 검증하도록 한다.
    - 이 때, `validate...` 메서드는 '검증의 목적'을 나타낸다. (ex_ `validateUpdatable`)
    - 단, '검증의 목적'을 나타내느라 직관성이 심하게 훼손될 경우, '무엇을 검증하는지'를 직관적으로 표현한다. (ex_ `validateNameNotNull`)
2. `validate...` 메서드는, 클래스 내부의 한 곳에 모아서 정렬한다. 클래스의 여러 곳에 흩어지도록 배치하지 않는다.
3. 동시성으로 인한 데이터베이스 제약조건 문제가 발생할 수 있는 기능이라면, `@Retryable`을 사용해 같은 검증 로직을 다시 수행하도록 구현한다.
    - 이를 통해, 레포지토리는 비즈니스 로직을 모르면서도, 데이터베이스 제약조건으로 인해 실패한 상황도 `try-catch` 블록 없이 서비스 검증 로직으로 처리할 수 있다.

## 레포지토리

레포지토리는 모든 비즈니스 요구사항에 의존하지 않는다.
순수한 DB 로직을 수행하도록 구현한다.
