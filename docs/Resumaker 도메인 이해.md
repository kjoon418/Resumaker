이 문서는, 프로젝트 'Resumaker'의 도메인 관점 명세를 제공하기 위해 작성되었다.

# 핵심 목적
핵심 목적 역시, 기획 과정에서 개선될 수 있다.

**상위 목적 (존재 이유)**
3. 취업 준비생이 이력서/포트폴리오를 작성할 때 겪는 핵심 페인포인트를 해결한다.

**수단 목적 (상위 목적을 실현하는 방법)**
1. 경험을 이력서/포트폴리오로 만드는 과정을 AI로 자동화한다. (MVP에서는 경험 기록→AI 생성→텍스트 산출물 흐름을 완결한다. 시각화·파일 내보내기는 로드맵.)
2. 사용자의 '백지의 공포'를 최소화한다. (경험을 적어나가는 과정을 정적 회상 보조로 능동적으로 돕고, 산출물은 AI가 초안을 만들어 빈 화면에서 시작하지 않게 한다.)
4. 경험/경력을 한 번만 잘 적으면, 회사별 추구하는 방향에 맞는 이력서/포트폴리오를 계속 만들어낼 수 있다.
5. 만들어진 이력서/포트폴리오를 쉽게 개선할 수 있다.

> 위계 근거: 목적 3(페인포인트 해결)이 서비스의 존재 이유다. 목적 1·2·4·5는 각각 자동화·공포 제거·재사용·개선이라는 수단으로 목적 3을 실현한다. 번호는 기존 식별자를 유지한다.

# 도메인 용어 정리
이 절에서 각 도메인의 영어 명칭을 한 번씩 드러낸 뒤, 이후 서술에서는 한글 명칭만 사용한다.

- **사용자(User):** 서비스에 가입해 자신의 경험을 기록하고 산출물을 만들어내는 취업 준비생.
- **경험 기록(Experience Record):** 사용자가 한 번 잘 적어두면 반복 재사용되는 원천 데이터의 최소 단위. 하나의 의미 있는 경험(프로젝트, 직무 활동, 대외활동, 수상, 학습 등) 하나를 담는다.
- **경험 묶음(Experience Library):** 한 사용자가 보유한 모든 경험 기록의 집합. 산출물 생성의 재료 창고 역할을 한다.
- **목표 정보(Target Brief):** 사용자가 겨냥하는 회사·직무·채용 방향을 담은 입력. 산출물을 어느 방향으로 다듬을지 결정한다.
- **이력서(Resume):** 경력·역량을 압축적·정형적으로 정리해, 채용 담당자가 짧은 시간에 적합성을 판단하도록 돕는 산출물.
- **포트폴리오(Portfolio):** 개별 경험을 서사와 근거 중심으로 풀어, 사용자가 무엇을 어떻게 해냈는지 깊이 있게 보여주는 산출물.
- **산출물(Artifact):** 이력서와 포트폴리오를 아우르는 상위 개념. AI 생성의 결과물.
- **생성 항목(Artifact Section):** 산출물을 이루는 재생성 가능한 부분 단위(예: 자기소개 문단, 한 개 경험의 서술 블록 등).
- **버전(Version):** 한 산출물의 한 시점 전체를 담은 산출물 단위 스냅샷. 사용자가 이전·이후를 비교·복원할 수 있게 한다. (구조·생명주기는 '산출물 구조와 상태 모델' 절 참고)
- **활성 버전(Active Version):** 한 산출물에서 현재 사용자가 열람·복사·비교의 기준으로 보는 버전. 채택을 통해 바뀐다.
- **항목 후보(Section Candidate):** 한 생성 항목을 재생성할 때 만들어지는 대안 결과. 사용자가 채택하기 전까지는 버전이 아니다.
- **생성 근거(Source Grounding):** 각 생성 항목이 어떤 경험 기록에서 비롯되었는지(항목 출처 연결), 그리고 산출물의 정량 수치·고유명사가 어떤 경험 기록의 어떤 문자열에 근거하는지(고정밀 사실 근거)를 잇는 두 층위의 연결. 신뢰성(환각 방지)의 핵심 장치. (구조는 '산출물 구조와 상태 모델' 절 참고)
- **이력서 양식(Resume Template):** 한 이력서가 어떤 섹션들을, 어떤 순서로, 어떤 성격으로 담을지 규정하는 구조·형식 정의. 무엇을 강조할지(목표 정보)가 아니라 어떤 칸에 담을지를 정한다. 목표 정보와 같은 격의 생성 입력이며, 사용자가 직접 지정하거나(프리셋 선택·회사 양식 붙여넣기) 미지정 시 AI가 경험에 맞춰 만든다.
- **섹션 정의(Section Definition):** 이력서 양식을 이루는 한 칸의 규격. 섹션 이름·역할, 성격(요약형/경력형), 필수·선택 여부를 담는다. 생성 시 이 정의 하나가 산출물의 생성 항목 하나(경력형은 하나 이상)로 실체화된다.
- **AI 생성 양식(AI-Derived Template):** 사용자가 양식을 지정하지 않았을 때 AI가 경험 묶음·목표 정보를 근거로 결정한 섹션 구조. MVP에서는 해당 산출물에 스냅샷으로만 보존되며, 재사용 가능한 독립 양식으로 라이브러리화하는 것은 로드맵.

# 핵심 가치 흐름 개요
서비스가 가치를 전달하는 기본 흐름은 다음과 같다.

1. **기록한다:** 사용자가 자신의 경험을 경험 기록 단위로 쌓아 경험 묶음을 만든다. (핵심 목적 2)
2. **겨냥한다:** 사용자가 목표 정보를 입력해, 어떤 회사·직무를 노리는지 알린다. (핵심 목적 4)
3. **생성한다:** AI가 경험 묶음과 목표 정보를 재료로, 목표에 최적화된 이력서/포트폴리오를 만든다. 이력서는 사용자가 지정한 양식(또는 미지정 시 AI가 정한 양식)에 따라 섹션이 구성된다. (핵심 목적 1)
4. **다듬는다:** 사용자가 마음에 들지 않는 부분을 항목 단위로 재생성하거나 직접 손본다. (핵심 목적 5)
5. **활용한다:** 사용자가 완성된 산출물을 열람하고 복사·내보내 실제 지원에 사용한다.
6. **재사용한다:** 동일한 경험 묶음으로 다른 목표 정보를 겨냥해 1~5를 반복한다. (핵심 목적 4)

이 흐름의 전제: **한 번 잘 기록한 경험은 영구히 저장되어 회사를 바꿔가며 무한히 재활용된다.** 이것이 계정 기반 다중 사용자 모델을 채택한 이유다.

# 기능별 상세 명세

## 1. 경험 기록
### 목적
산출물의 모든 재료가 되는 원천 데이터를, 사용자가 백지의 공포 없이 구조적으로 쌓게 한다.

### 이뤄낼 가치
- 사용자는 "이력서에 뭘 써야 하지"가 아니라 "내가 뭘 했지"만 떠올리면 된다. 글쓰기가 아니라 회상에 집중하게 만든다. (핵심 목적 2 — 정적 회상 보조로 능동적으로 실현)
- 한 번 기록한 경험은 회사·직무가 바뀌어도 재사용되므로, 기록 비용을 단 한 번만 지불한다. (핵심 목적 4)

### 도메인 단위와 구조 (결정)
하나의 경험 기록은 다음으로 구성된다.

**필수 항목**
- **제목:** 이 경험을 한눈에 식별하는 짧은 이름. (예: "사내 결제 시스템 개편 프로젝트")
- **경험 유형:** 프로젝트 / 직무·인턴 / 대외활동 / 수상·자격 / 학습·교육 중 하나. 산출물 생성 시 맥락 판단에 쓰인다.
- **본문(무엇을 했는가):** 사용자가 자유 서술로 적는, 그 경험에서 실제로 한 일. 이 한 칸이 비어 있으면 생성의 재료가 없으므로 필수다.

**선택 항목 (있으면 산출물 품질이 올라가는 재료)**
- **상황·과제(Situation):** 어떤 맥락·문제 상황이었는가.
- **행동(Action):** 그 안에서 내가 구체적으로 한 행동·역할.
- **결과(Result):** 정량·정성적 성과. (예: "응답 속도 40% 개선")
- **기간:** 시작·종료 시점.
- **사용 역량·기술:** 이 경험에서 발휘한 기술·역량 태그.

> 도메인 결정 근거: 필수 항목을 '제목·유형·본문' 3개로 최소화해 기록 진입 장벽을 낮춘다(핵심 목적 2). 동시에 상황·행동·결과(STAR) 구조를 선택 항목으로 안내해, 채우면 산출물 품질이 올라가는 점진적 구조화를 유도한다.

### 정적 회상 보조 (MVP 포함, AI 비용 무관)
목적 2("백지의 공포 최소화")를 MVP에서 온전히 실현하기 위해, AI 호출 없이 정적 텍스트만으로 사용자의 회상을 능동적으로 돕는다.

**본문 칸 예시 플레이스홀더:** 본문(무엇을 했는가) 칸이 비어 있을 때, 다음과 같은 예시 문구를 흐릿하게 표시해 사용자가 무엇을 적어야 하는지 즉시 감을 잡게 한다.
> 예시: "결제 시스템의 응답 속도 문제를 발견하고, 캐싱 전략을 도입해 40% 개선했습니다."

**유도 질문 세트:** 본문 칸 근처에 다음 질문들을 정적 텍스트로 제공한다. 사용자는 이 질문들에 답하듯 적으면 된다.
- 그 경험에서 내가 직접 한 일은 무엇인가?
- 어떤 문제나 목표가 있었는가?
- 결과적으로 무엇이 달라졌는가? (수치가 없어도 괜찮다)

> 도메인 결정 근거: AI 인터뷰형 보강은 LLM 호출이 필요해 비용 가드레일과 연동되므로 로드맵으로 미룬다. 정적 회상 보조는 비용 없이 즉시 MVP에 포함 가능하며, 목적 2를 절반이 아닌 온전히 닫는다. (P1 페인포인트 직접 해소)

### 사용자 시나리오
- **성공:** 사용자가 '경험 추가'를 눌러 제목·유형·본문을 적고 저장한다. 경험이 경험 묶음에 추가되고, 이후 모든 산출물 생성에서 재료로 쓰인다.
- **성공(점진 보강):** 처음엔 본문만 적었다가, 나중에 상황·행동·결과를 채워 같은 경험을 더 풍부하게 만든다.
- **실패(필수 누락):** 본문 없이 저장을 시도하면, 차단하고 "이 경험에서 무슨 일을 했는지 한 줄이라도 적어 주세요. 나중에 더 자세히 보강할 수 있어요." 식으로 안내한다. (UX 에러 가이드: 해결 방법 제시 + 부정 감정 최소화)
- **실패(빈 경험 묶음):** 경험이 하나도 없는 사용자가 곧장 산출물 생성으로 가려 하면, 생성을 막고 경험 기록부터 유도한다. (아래 5번 산출물 생성의 실패 케이스와 연결)

### 수용 기준
- 제목·유형·본문 중 하나라도 비면 저장이 거부되고, 어느 항목이 비었는지 사용자가 알 수 있다.
- 저장된 경험 기록은 동일 사용자의 이후 산출물 생성 후보 재료 목록에 100% 나타난다.
- 선택 항목은 비어 있어도 저장이 성공한다.

## 2. 목표 정보 입력
### 목적
같은 경험 묶음이라도 회사·직무에 따라 다른 산출물이 나오도록, 사용자가 겨냥하는 방향을 명시하게 한다.

### 이뤄낼 가치
- 한 벌의 경험으로 회사마다 최적화된 산출물을 뽑아내는 재사용성을 실현한다. (핵심 목적 4)
- 사용자가 "이 회사엔 뭘 강조해야 하지"를 고민하는 부담을 AI에게 넘긴다. (핵심 목적 3)

### 입력 형태 (결정)
MVP에서 목표 정보는 사용자가 직접 입력한다.
- **회사명·직무명:** 짧은 식별 정보. (선택)
- **채용 방향 텍스트:** 채용공고 본문, 또는 사용자가 정리한 "이 회사가 원하는 인재상·요구 역량" 자유 서술. (필수)

> 도메인 결정 근거: 채용공고 자동 연동(크롤링·API)은 외부 의존성과 법적 리스크가 크므로 로드맵으로 미룬다. MVP는 "사용자가 공고 텍스트를 붙여넣는다"는 가장 단순하고 확실한 형태를 택한다.

### 사용자 시나리오
- **성공:** 사용자가 산출물 생성 직전 채용 방향 텍스트를 입력(붙여넣기)하면, 그 방향이 생성에 반영된다.
- **성공(저장·재사용):** 자주 지원하는 회사의 목표 정보를 저장해 두고, 다음 생성 때 다시 고른다.
- **실패(빈 방향):** 채용 방향 텍스트 없이 생성하려 하면, 차단하고 "어떤 회사·직무를 겨냥하는지 알려주시면 그 방향에 맞춰 만들어 드려요. 공고 내용을 붙여넣어도 좋아요." 식으로 안내한다.
- **실패(무관한 목표):** 입력된 목표 방향과 경험 묶음이 동떨어진 경우(예: 디자인 경험만 있는데 회계 직무를 겨냥), 생성은 진행하되 "겨냥하신 직무와 직접 맞닿는 경험이 적어요. 관련 경험을 더 추가하면 결과가 좋아져요." 식으로 사용자에게 한계를 미리 알린다. (날조로 빈자리를 채우지 않는다 — 신뢰성 가드레일과 연결)

### 수용 기준
- 채용 방향 텍스트가 비면 생성이 거부되고 그 이유가 안내된다.
- 동일 목표 정보로 생성하면, 다른 목표 정보로 생성한 결과와 강조점이 달라진다(같은 경험 묶음 기준, 사람이 비교해 차이를 식별 가능).

## 2.5 이력서 양식 지정
### 목적
회사가 정해진 이력서 양식을 요구하는 경우가 잦으므로, 사용자가 산출물의 섹션 구조·순서를 직접 정하거나, 정하지 않으면 AI가 경험에 맞춰 최적 구조를 제공하게 한다.

### 이뤄낼 가치
- 회사가 요구하는 양식에 맞춰 그대로 제출할 수 있는 이력서를 만든다(P2 "회사마다 매번 새로" 완화 — 구조 차원). (핵심 목적 4)
- 양식 고민이 없는 사용자는 AI가 정한 구조에서 바로 시작한다(백지의 공포 — 구조 차원). (핵심 목적 2)

### 양식의 도메인 위치와 직교성 (결정)
- 이력서 양식은 목표 정보와 **같은 격의 생성 입력**이며 서로 **직교한다**. 목표 정보는 "무엇을 강조하나(내용 방향)"를, 양식은 "어떤 섹션에 담나(구조·형식)"를 정한다. 같은 목표에 다른 양식, 같은 양식에 다른 목표를 자유로이 조합할 수 있으므로 둘을 분리한다.
- 양식은 목표 정보와 동일하게 **사용자 소유로 저장·재사용**된다(사용자 1 — N 양식). 회사마다 정해진 양식을 한 번 만들어 두고 지원처를 바꿔가며 재선택한다. 저장된 양식의 수정·삭제 정책은 목표 정보의 저장·삭제 정책을 그대로 따른다.

### 양식의 구성요소 (결정)
- 양식은 **섹션 정의의 순서 있는 목록**이다. 각 섹션 정의는 다음을 담는다.
    - **섹션 이름·역할:** 그 칸이 무엇을 담는지(예: "한 줄 자기소개", "핵심 역량", "주요 경력").
    - **섹션 성격:** 요약형(여러 경험 종합, 경험과 N:M) 또는 경력형(하나 이상 경험에 근거). '산출물 구조와 상태 모델' 절의 생성 항목 2종과 1:1 대응한다.
    - **필수·선택:** 회사 양식이 특정 섹션을 강제하는 경우를 표현한다. 근거 경험이 없을 때 동작이 달라진다(아래 시나리오·신뢰성 가드레일 참고).
- 양식은 **구조·형식만** 담고 사실 내용은 담지 않는다(신뢰성 가드레일 절 참고).

### 사용자 지정 양식의 입력 형태 (결정)
MVP에서 사용자 지정 양식은 두 경로를 제공한다.
- **프리셋 선택:** 서비스가 제공하는 표준 이력서 양식 몇 종 중 하나를 고른다. (AI 호출 없음, 즉시)
- **회사 양식 붙여넣기:** 회사가 요구한 양식 설명·항목 목록 텍스트를 붙여넣으면 AI가 이를 섹션 정의 목록으로 해석한다. 해석 결과는 **사용자가 확인·확정한 뒤에만** 양식으로 확정되며, 부적절하면 프리셋이나 AI 생성 양식으로 폴백한다. (사용자가 섹션을 직접 추가·삭제·정렬하는 자유 편집은 로드맵.)
- 한 번의 생성에는 하나의 양식만 적용한다.

> 도메인 결정 근거: §2 목표 정보가 "가장 단순·확실한 형태=붙여넣기"를 택한 것과 동일하게, 양식도 폼빌더 같은 복잡한 직접 편집 대신 프리셋·붙여넣기를 MVP로 택한다. 붙여넣기 해석은 비결정적 AI 추론이므로, 사용자 확정 게이트 뒤로 격리해 "사용자가 의도한 양식이 반영됐는가"를 결정적으로 판정 가능하게 한다.

### AI 생성 양식 (기본값, 결정)
- 사용자가 양식을 지정하지 않으면, AI가 경험 묶음 구성과 목표 정보를 근거로 섹션 구조(AI 생성 양식)를 결정한 뒤 그 구조로 이력서를 생성한다.
- MVP에서 AI 생성 양식은 **해당 산출물에 스냅샷으로 보존**된다(아래 '양식과 생성 항목의 대응' 참고). AI 생성 양식을 독립적·재사용 가능한 양식으로 저장·노출하는 것은 로드맵.

### 사용자 시나리오
- **성공(프리셋):** 사용자가 이력서 생성 직전 프리셋 양식을 고르면, 그 섹션 구조로 초안이 만들어진다.
- **성공(붙여넣기):** 사용자가 회사 양식 텍스트를 붙여넣으면, AI가 섹션 정의로 해석하고 사용자가 확정한 뒤 그 구조로 생성한다.
- **성공(미지정):** 사용자가 아무 양식도 고르지 않고 생성하면, AI가 경험에 맞춰 구조를 정하고 그 구조가 산출물에 스냅샷으로 남는다.
- **실패(붙여넣기 해석 불가·빈 입력):** 붙여넣은 텍스트에서 섹션 구조를 해석할 수 없으면, 차단하지 않고 "양식을 알아보기 어려워요. 프리셋을 고르거나, AI가 알아서 구조를 정하게 둘 수 있어요." 식으로 폴백을 제시한다.
- **실패(필수 섹션 근거 없음):** 사용자가 필수로 지정한 섹션에 뒷받침 경험이 없으면, 빈 섹션을 날조로 채우지 않는다. 생성은 진행하되 "이 섹션을 채우려면 관련 경험을 추가하세요"를 산출물 상단에 명확히 고지한다(차단하지 않음 — §2 무관한 목표 정책과 일관). 선택 섹션은 근거가 없으면 조용히 생략한다. (신뢰성 가드레일과 연결)

### 수용 기준
- 사용자가 프리셋·붙여넣기로 지정한 양식의 각 필수 섹션 정의에, 근거 경험이 있는 한, 대응하는 생성 항목이 산출물에 1:1로 존재한다(결정적 대조).
- 붙여넣기로 만든 양식은 AI 해석 결과를 사용자가 확정한 뒤에만 생성에 적용된다.
- 사용자가 양식을 지정하지 않으면 AI 생성 양식으로 생성되고, 그 구조가 산출물에 스냅샷으로 보존되어 이후 열람 시 동일하게 나타난다.
- 필수 섹션의 근거 경험이 0이면 그 섹션은 날조로 채워지지 않고, 생성은 진행되며 부족이 고지된다. 선택 섹션의 근거 0은 생략된다.

## 3. 이력서 생성
### 목적
경험 묶음과 목표 정보를 재료로, 목표에 최적화된 이력서를 AI가 만들어 백지 작성 부담을 없앤다.

### 이뤄낼 가치
- 사용자는 빈 화면이 아니라 "다듬을 초안"에서 시작한다. (핵심 목적 1, 2)
- 목표 정보에 맞춰 강조점이 자동 조정된다. (핵심 목적 4)

### 이력서의 도메인 성격 (결정)
이력서는 **압축적·정형적** 산출물이다.
- 독자: 짧은 시간에 다수 지원자를 훑는 채용 담당자.
- 형태: 핵심 역량·경력을 요약한 정형 구조(예: 한 줄 자기소개, 핵심 역량, 주요 경력 요약 항목들). 이 예시는 기본 프리셋 양식의 섹션 구성이며, 실제 섹션 집합은 적용된 양식(2.5 참고)에 따라 달라진다.
- 생성 방식: 경험 묶음에서 목표 정보와 관련도가 높은 것을 골라, 짧고 임팩트 있게 압축한다.

### 1차 생성 단위 (결정)
- **이력서는 전체 단위로 1차 생성**하되, **결과는 적용된 양식의 섹션 정의 각각에 대응하는 생성 항목들로 나뉘어** 제시된다. 어떤 생성 항목들이 만들어지는지는 더 이상 예시가 아니라 **적용된 양식(사용자 지정 또는 AI 생성)이 결정한다**(2.5 및 '양식과 생성 항목의 대응' 참고). 사용자는 이후 항목별로 재생성한다. (재생성은 4번 참고)

### 사용자 시나리오
- **성공:** 사용자가 사용할 경험들을 고르고(또는 전체) 목표 정보를 확인한 뒤 '이력서 생성'을 누르면, 항목으로 구성된 이력서 초안 한 벌이 만들어지고 새 버전으로 저장된다.
- **실패(재료 부족):** 경험 묶음이 비었거나 너무 빈약하면 생성을 차단하고 경험 기록을 유도한다. (예방형 에러)
- **실패(부분 생성 실패):** 일부 항목 생성에 실패하면 전체를 버리지 않는다. 성공한 항목은 보여주고, 실패한 항목만 표시하며 "이 부분만 다시 만들어 볼게요" 식의 재시도 경로를 준다. (UX 에러 가이드: 회복 경로 제공)
- **실패(생성 한도 초과):** 비용 가드레일 상한에 도달하면, 남은 횟수·회복 시점을 사용자 언어로 안내한다. (가드레일 절 참고)

### 수용 기준
- 생성된 이력서의 모든 사실 진술은 경험 묶음 안에 근거가 존재한다(생성 근거로 추적 가능). 근거 없는 새 사실이 0건이어야 한다.
- 생성 결과는 즉시 하나의 버전으로 저장되어, 이후 열람·비교가 가능하다.
- 빈 경험 묶음으로는 이력서가 생성되지 않는다.

## 4. 포트폴리오 생성
### 목적
개별 경험을 서사·근거 중심으로 깊이 있게 풀어, 사용자의 역량을 입체적으로 보여주는 포트폴리오를 만든다.

### 이뤄낼 가치
- 이력서가 담지 못하는 '어떻게·왜 해냈는가'의 깊이를 전달한다. (핵심 목적 1, 3)
- 같은 경험을 회사 방향에 맞춰 다른 서사로 재구성한다. (핵심 목적 4)

### 포트폴리오의 도메인 성격 (결정)
포트폴리오는 **서사적·심화적** 산출물이다.
- 독자: 지원자를 깊이 검토하려는 실무자·면접관.
- 형태: 경험별로 상황→행동→결과→배운 점을 풀어 쓴 서술 묶음.
- **MVP에서 포트폴리오는 '텍스트 구조물'이다.** 시각 디자인(레이아웃·이미지·테마)은 로드맵으로 미룬다.

> 도메인 결정 근거: 시각 디자인 산출물은 핵심 목적(경험→이력서/포트폴리오 자동화)의 본질이 아니라 표현 계층이다. MVP는 "내용을 잘 뽑아준다"는 본질에 집중하고, 시각화는 가치가 검증된 뒤 더한다. 이력서와 포트폴리오의 차이는 '시각화 여부'가 아니라 '압축 vs 서사'라는 내용적 차이로 정의한다.

### 1차 생성 단위 (결정)
- 포트폴리오는 **선택된 경험들 각각을 하나의 생성 항목으로** 1차 생성한다. 경험 단위로 묶이므로 항목별 재생성이 자연스럽다.

### 사용자 시나리오
- **성공:** 사용자가 포트폴리오에 넣을 경험들을 고르고 목표 정보를 확인한 뒤 생성하면, 경험별 서술 항목으로 이뤄진 포트폴리오 한 벌이 새 버전으로 만들어진다.
- **실패(재료 부족·부분 실패·한도 초과):** 이력서 생성과 동일한 회복 정책을 따른다.

### 수용 기준
- 포트폴리오의 각 서술 항목은 정확히 하나 이상의 경험 기록에 근거한다(생성 근거 추적 가능).
- 한 경험에 대한 서술은, 동일 경험을 이력서에서 압축한 것보다 더 길고 서사적이다(사람이 비교해 차이 식별 가능).

## 5. 개선/재생성
### 목적
완성된 산출물을 통째로 다시 만들지 않고, 마음에 들지 않는 부분만 골라 손쉽게 개선한다.

### 이뤄낼 가치
- 만들어진 산출물을 적은 비용으로 원하는 형태에 수렴시킨다. (핵심 목적 5)
- 한 번의 마음에 안 드는 결과가 전체를 버리는 비용으로 이어지지 않게 한다.

### 개선 단위 (결정)
개선은 두 경로를 제공한다.
- **항목 재생성:** 특정 생성 항목만 AI로 다시 만든다. 사용자는 같은 항목의 여러 후보·버전을 비교해 고른다.
- **직접 편집:** 사용자가 항목 텍스트를 직접 수정한다. AI 생성 결과는 어디까지나 초안이며 최종 결정권은 사용자에게 있다.
- **개선 지시:** 항목 재생성 시 "더 짧게", "성과 수치를 강조" 같은 방향 지시를 덧붙일 수 있다.

### 버전·비교 (결정)
- 재생성·편집은 새 버전을 만들고 이전 버전을 보존한다.
- 사용자는 한 산출물의 버전들을 비교해 더 나은 쪽을 채택할 수 있다.
- 버전 보관 개수에는 상한이 있다(가드레일 절 참고). 상한 초과 시 가장 오래된 버전부터 정리하며, 정리 전에 사용자에게 알린다.

### 사용자 시나리오
- **성공(항목 재생성):** 자기소개 문단이 마음에 안 들어 그 항목만 재생성하면, 다른 항목은 그대로 둔 채 새 후보가 나오고 사용자가 채택한다.
- **성공(직접 편집·비교):** AI 결과를 직접 다듬고, 마음이 바뀌면 이전 버전과 비교해 되돌린다.
- **실패(재생성 한도 초과):** 항목 재생성 한도 초과 시 막고, "오늘 다시 만들 수 있는 횟수를 다 썼어요. 직접 다듬어 보거나 내일 이어서 해요." 식으로 대안을 제시한다.
- **실패(개선 지시 무관):** 경험 근거로 뒷받침되지 않는 지시(예: 없는 성과를 넣어 달라)는 따르지 않고, 그 이유를 알린다. (신뢰성 가드레일)

### 수용 기준
- 한 항목을 재생성해도 다른 항목 내용은 변하지 않는다.
- 재생성·편집 후 직전 버전이 보존되어 비교·복원이 가능하다(보관 상한 이내).
- 재생성으로도 경험 근거 밖의 새 사실은 추가되지 않는다.

---

## 6. 결과물 활용
### 목적
생성된 산출물이 화면 안에 머물지 않고, 사용자가 실제 지원에 쓸 수 있게 한다. 가치 흐름의 종착점.

### 이뤄낼 가치
- "만들었다"에서 "실제로 썼다"로 가치 흐름을 완결한다.

### 활용 범위 (결정)
- **MVP: 열람 + 복사.** 사용자는 산출물을 화면에서 보고, 전체 또는 항목 단위로 텍스트를 복사해 원하는 곳(지원 폼·문서 도구)에 붙여 쓴다.
- **로드맵: 파일 내보내기(PDF/문서).** 시각 디자인 로드맵과 함께 다룬다.

> 도메인 결정 근거: MVP의 산출물이 '텍스트 구조물'이므로, 텍스트 복사만으로도 실사용 가치가 완결된다(핵심 목적 1, 5). PDF 등 서식 있는 내보내기는 시각화 로드맵에 종속되므로 함께 미룬다. 이로써 MVP가 "내용을 잘 만든다"는 본질에 집중하도록 경계를 긋는다.

### 사용자 시나리오
- **성공:** 사용자가 완성된 산출물을 열람하고, '전체 복사' 또는 항목별 복사로 텍스트를 가져가 지원에 사용한다.
- **실패(복사 미지원 환경):** 복사가 막힌 환경이면, 사용자가 직접 선택·복사할 수 있도록 텍스트를 그대로 노출해 막다른 길을 만들지 않는다.

### 수용 기준
- 저장된 산출물은 생성 이후 언제든 다시 열람된다.
- 산출물 전체 및 각 항목 텍스트를 복사로 가져올 수 있다.

## 7. 계정/데이터 소유
### 목적
경험을 한 번 잘 적어 영구 재사용한다는 핵심 전제를 계정 단위로 보장하고, 사용자가 자기 데이터의 운명을 통제하게 한다.

### 이뤄낼 가치
- 경험 묶음·산출물이 사용자 소유로 영구 보존되어 회사를 바꿔가며 재활용된다. (핵심 목적 4)
- 사용자가 자기 데이터를 신뢰하고 떠날 권리(삭제)도 갖는다.

### 도메인 규칙 (결정)
- 모든 경험 기록·목표 정보·산출물·버전은 정확히 한 명의 사용자에게 귀속된다. 다른 사용자는 접근할 수 없다.
- 가입·로그인 후 사용자의 데이터는 명시적으로 지우기 전까지 영구 보존된다.
- **계정 삭제 시:** 그 사용자의 모든 경험 기록·목표 정보·산출물·버전이 함께 삭제된다. 사용자에게 '되돌릴 수 없음'을 명확히 고지하고 확인을 받는다.
- **경험 기록 삭제 시:** 이미 만들어진 산출물 버전은 생성 시점의 스냅샷이므로 그대로 보존된다(과거 산출물이 갑자기 무너지지 않게). 단, 이후의 새 생성에서는 그 경험이 더 이상 재료로 쓰이지 않는다.

### 사용자 시나리오
- **성공(재방문):** 사용자가 다시 로그인하면 이전 경험 묶음·산출물이 그대로 있어 곧바로 새 목표를 겨냥할 수 있다.
- **성공(계정 삭제):** 사용자가 탈퇴를 요청하면, 무엇이 사라지는지 안내받고 확인 후 데이터가 완전히 삭제된다.
- **실패(권한 없는 접근):** 다른 사용자의 데이터에 접근하려는 시도는 차단된다.

### 수용 기준
- 한 사용자의 데이터는 다른 사용자에게 절대 노출되지 않는다.
- 로그아웃 후 재로그인해도 기존 경험 묶음·산출물이 동일하게 보존된다.
- 계정 삭제 후 해당 사용자의 모든 귀속 데이터가 조회되지 않는다.

# 산출물 구조와 상태 모델
이 절은 산출물·생성 항목·버전·생성 근거의 관계와 생명주기를 결정적으로 정의한다. 개발자가 도메인 모델(엔티티·관계·상태 전이)을 추측 없이 도출할 수 있게 하는 것이 목적이다.

## 산출물 · 생성 항목 · 버전의 관계 (결정)
- 하나의 산출물(이력서 또는 포트폴리오)은 여러 생성 항목으로 구성된다.
- **버전은 산출물 단위의 스냅샷이다.** 한 버전은 그 시점 산출물을 이루는 모든 생성 항목의 내용을 통째로 담는다.
- 산출물은 항상 하나의 활성 버전을 가리킨다. 사용자가 열람·복사·비교의 기준으로 보는 것이 활성 버전이다.
- 항목을 재생성하거나 직접 편집해 채택하면, 직전 활성 버전을 복제한 위에 그 항목만 교체한 새 버전이 만들어지고 활성 버전이 그 버전으로 바뀐다. 교체되지 않은 항목은 직전 버전에서 그대로 복제된다(그래서 한 항목 변경이 다른 항목을 바꾸지 않는다).
- 카디널리티: 사용자 1 — N 산출물, 산출물 1 — N 버전, 버전 1 — N 생성 항목.

> 결정 근거: 버전을 산출물 단위 스냅샷으로 고정하면 "경험 기록 삭제 후에도 과거 산출물이 무너지지 않는다"(기능 7)와 "버전 비교·복원"(기능 5)이 한 모델로 일관되게 설명된다. 항목 단위 독립 버전 체인은 비교·복원의 의미를 흐려 채택하지 않는다.

## 항목 후보와 채택 (결정)
- 항목 재생성은 그 항목의 후보를 만든다. 후보는 채택 전까지 버전이 아니며, 사용자는 여러 후보를 비교해 하나를 고른다.
- 채택은 상태 전이다: 후보(또는 직접 편집 결과)를 활성으로 확정해 새 버전을 만든다.

## 생성 항목의 종류와 경험 카디널리티 (결정)
- 이력서 생성 항목은 두 종류다.
    - 요약형 항목(예: 한 줄 자기소개, 핵심 역량 요약): 여러 경험 기록을 종합해 만들어지며 특정 경험 하나에 매이지 않는다(경험과 N:M).
    - 경력 항목(주요 경력 요약): 하나 이상의 경험 기록에 근거한다.
- 포트폴리오 생성 항목은 선택된 경험 기록과 1:1로 대응한다.

## 양식과 생성 항목의 대응 (결정)
- 이력서 산출물은 생성 시점에 적용된 양식의 **불변 스냅샷 하나**를 가지며, 그 산출물의 **모든 버전이 이 스냅샷을 공유**한다. 버전은 여전히 항목 내용의 스냅샷만 담는다(앞의 버전 정의 불변). 카디널리티: 산출물 1 — 1 양식 스냅샷, 재사용 가능 양식 1 — N 산출물.
- 이력서 한 버전이 담는 생성 항목 집합은, 그 산출물의 양식 스냅샷에 담긴 섹션 정의 중 **근거 경험이 있는 것들**로 결정된다. 요약형 섹션 정의는 요약형 생성 항목 하나로, 경력형 섹션 정의는 하나 이상의 경력 항목으로 실체화된다.
- **근거 경험이 0인 섹션 정의는 생성 항목으로 실체화하지 않는다.** 필수 섹션이면 그 부족을 고지하고(생성은 진행), 선택 섹션이면 생략한다. 이로써 "근거 없이 만들어진 항목 0건"(신뢰성 가드레일)과 "항목 집합이 양식으로 결정된다"가 양립한다.
- **버전 비교에서 '같은 항목'은 같은 섹션 정의에 대응하는 항목으로 본다.** 한 산출물의 버전들은 동일한 양식 스냅샷을 공유하므로 섹션 정의를 키로 버전 간 항목이 안정적으로 대응된다.
- 항목 재생성은 그 항목이 속한 섹션 정의를 바꾸지 않는다(구조 불변, 내용만 갱신). 양식 자체를 바꾸는 것은 다른 생성 입력으로 새로 만드는 것이므로, 목표 정보를 바꿔 재생성하는 것과 동일하게 **새 산출물 생성**으로 다룬다(MVP).
- 포트폴리오에는 양식이 적용되지 않는다. 포트폴리오 생성 항목은 경험당 하나로 고정되어 선택 경험 기록과 1:1로 대응하므로(앞의 결정 유지), 섹션 구조를 고를 여지 자체가 없다.

> 결정 근거: 양식 스냅샷의 소유 단위를 산출물로 고정하면 버전 모델(버전=항목 내용 스냅샷)을 건드리지 않고도 "버전 1—N 생성 항목"이 예시가 아니라 결정적으로 정의된다(이 절의 목적). 양식을 목표 정보와 동급 생성 입력으로 두면 "양식을 바꾼 비교"가 "목표 정보를 바꾼 비교"와 동일하게 별개 산출물로 일관 설명되어, 별도의 버전 비교 차원을 도입하지 않는다.

## 생성의 정합성 단위와 부분 실패 (결정)
- **생성의 성공/실패 판정 단위는 생성 항목이다.** 각 항목은 독립적으로 성공·실패한다.
- 1차 생성은 대상 항목 전체의 생성을 시도하고, 성공 항목과 실패 항목을 함께 담은 하나의 버전으로 저장된다. 즉 부분 실패 버전도 정식 버전이며, 사용자는 성공분을 잃지 않는다.
- 각 생성 항목은 상태를 가진다: **생성중 / 생성성공 / 생성실패(재시도 대기) / 검증실패(재시도 대기)**.
- 실패·검증실패 항목은 사용자가 그 항목만 재시도하며, 재시도가 성공하면 채택을 통해 새 버전이 된다.
- 동시 재생성: 한 생성 항목은 동시에 하나의 진행 중 생성만 가진다. 같은 항목에 대한 중복 재생성 요청은 거절하고 진행 중임을 알린다. (서로 다른 항목은 병렬 재생성 가능.)

> 결정 근거: 외부 AI 호출은 실패 확률이 높으므로(Spring 트랜잭션 분리 가이드), 정합성 단위를 항목으로 좁혀 한 항목의 실패가 전체 생성을 무효화하지 않게 한다. 이는 "부분 실패 시 성공분 보존"(기능 3·4)과 정합한다.

## 생성 근거의 구조 (결정)
생성 근거는 두 층위로 분리해 추적한다.
- **항목 출처 연결(층위 1):** 각 생성 항목은 자신이 어떤 경험 기록들에서 비롯되었는지 출처 경험 기록 목록을 가진다(항목 1 — N 경험 기록). 사용자가 "이 문장이 어느 경험에서 나왔는지" 확인하고 신뢰하게 하는 탐색용 연결이며, 생성 항목에 속한 값으로 본다.
- **고정밀 사실 근거(층위 2):** 산출물에 등장하는 정량 수치·고유명사 각각은, 그것을 뒷받침하는 경험 기록과 그 안의 근거 문자열에 연결된다(사실 토큰 1 — 1 근거). 신뢰성 자동 검증(아래)의 대상이다.
- **생성 근거의 산출 주체·시점:** 생성 근거는 AI가 산출물을 생성하는 바로 그 시점에 함께 산출한다. 각 항목의 출처 경험 식별자와 고정밀 사실의 근거를 생성 결과와 함께 구조적으로 받는다. 생성 후 별도의 추론으로 근거를 역추적하지 않는다(검증에 비결정성을 추가하지 않기 위함).

> 결정 근거: 두 층위를 분리하면 "항목→경험"(탐색·신뢰)과 "사실 토큰→근거 문자열"(자동 검증)이라는 서로 다른 입도의 요구가 모순 없이 공존한다. 근거를 생성 시점에 확정하면 검증이 결정적이고 추가 AI 호출이 필요 없다.

# 비용·신뢰성 가드레일

## 비용 가드레일
### 목적
AI 생성은 호출당 비용이 발생하므로, 무제한 사용으로 인한 비용 폭증과 어뷰징을 도메인 정책으로 막는다.

### 정책 (결정)
카운트 대상·차감 시점·리셋 경계는 도메인 규칙으로 고정하고, 구체 수치만 운영 정책으로 분리한다.
- **1차 생성 횟수 상한:** 사용자당 카운트한다. 리셋 경계는 사용자 시간대 기준 달력일(자정 리셋)이다("내일 이어서"라는 안내와 일치). 1차 생성은 최소 한 항목이라도 성공하면 1회로 차감하고, 전 항목이 실패하면 차감하지 않는다.
- **항목 재생성 횟수 상한:** 생성 항목당 카운트한다(한 항목을 몇 번까지 다시 만들 수 있는가). 사용자가 요청한 재생성은 최종 성공 시 1회 차감하며, 그 과정에서 검증실패로 발생한 자동 재시도(신뢰성 가드레일 참고)는 별도로 차감하지 않는다.
- **버전 보관 개수 상한:** 산출물당 보관 버전 수에 상한을 둔다. 초과 시 가장 오래된 버전부터 정리하며 사전 고지한다.
- 상한 도달 시 작업을 막되, 사용자 언어로 남은 횟수·회복 시점·대안(직접 편집 등)을 안내한다. (UX 에러 가이드)

> 구체적 수치(하루 N회, 항목당 N회, 버전 N개 등)는 운영·비용 데이터로 조정될 값이므로 이 문서는 카운트 대상·차감 시점·리셋 경계만 고정하고, 수치는 운영 정책으로 분리한다.

### 수용 기준
- 각 상한에 도달하면 해당 작업이 차단되고 그 사실·대안이 사용자에게 안내된다.
- 버전 보관 상한 초과 시 가장 오래된 버전부터 정리되며, 정리 전 사용자에게 고지된다.

## 신뢰성 가드레일 (환각 방지)
### 목적
이력서의 생명은 신뢰성이다. AI가 사용자가 적지 않은 경험·성과를 날조하거나 과장하면, 면접에서 무너지고 서비스 신뢰가 깨진다.

### 정책 (결정)
- **근거 없는 사실 금지:** 산출물의 모든 사실 진술(경력·성과·역량)은 경험 묶음 안에 근거가 있어야 한다. 없는 경험·수치를 지어내지 않는다.
- **표현 다듬기는 허용, 사실 창작은 금지:** AI는 사용자가 적은 내용을 더 설득력 있게 '표현'할 수 있으나, 사용자가 적지 않은 '사실'을 추가할 수 없다.
  - **허용 예시:** "응답 속도를 줄였다" → "응답 속도를 단축해 사용자 경험을 개선했다" (표현 강화, 사실 동일)
  - **금지 예시:** "응답 속도를 줄였다" → "응답 속도를 40% 단축했다" (사용자가 적지 않은 수치를 AI가 삽입)
- **고정밀 항목 규칙:** 정량 수치(%, ms, 건수 등), 고유명사(회사명·기술명·프로젝트명), 성과 주장("최우수상 수상", "MAU 10만 달성" 등)은 사용자 기록에 **문자적 근거**가 있을 때만 산출물에 등장한다. 유사한 수치로 '추정'하거나 일반적인 사례에서 가져오는 것도 금지한다.
- **빈자리는 채우지 않고 알린다:** 목표에 필요한 경험이 부족하면 날조로 메우지 않고, 사용자에게 "이 부분을 강화하려면 관련 경험을 추가하세요"라고 안내한다.
- **양식은 구조만 규율하고 사실을 강제하지 않는다:** 사용자가 지정한 양식(특히 회사 양식 붙여넣기)이 특정 섹션을 요구해도, 그 섹션을 뒷받침할 경험 근거가 없으면 날조로 채우지 않는다. 빈 섹션은 채우지 않고 "이 섹션을 채우려면 관련 경험을 추가하세요"라고 알린다(필수 섹션은 고지 후 진행, 선택 섹션은 생략 — 2.5 참고). 양식은 어떤 칸이 있는지를 정할 뿐 그 칸의 사실 내용을 보증하지 않으며, 근거 없는 사실 금지 정책이 양식 요구보다 우선한다.
- **개선 지시의 한계:** 사용자의 개선 지시라도 근거 없는 사실 추가를 요구하면 따르지 않고 이유를 알린다.

### 자동 검증 규칙 (결정 — 결정적·구현 가능)
"근거 없는 사실 0건"을 개발자가 결정적으로 판정할 수 있도록 자동 검증의 범위와 방식을 한정한다.
- **자동 실패 판정 대상은 정량 수치와 고유명사 두 범주뿐이다.** 이 둘은 산출물에서 추출한 뒤, 그 토큰이 해당 항목의 출처 경험 기록 본문에 문자적으로 존재하는지 대조해 결정적으로 판정할 수 있다.
- **판정 주체는 결정적 검사(추출 + 문자열 대조)다.** 또 다른 AI 판정(LLM 기반 심사)을 자동 실패 판정에 쓰지 않는다(검증 단계에 비결정성을 들이지 않기 위함).
- **추출은 산출물에서 독립적으로 한다.** 자동 검증은 AI가 신고한 층위2 근거를 신뢰하지 않고, 산출물 텍스트에서 수치·고유명사를 직접 추출해 출처 경험 기록과 대조한다. (AI가 토큰을 누락 신고해 검증 구멍이 생기는 것을 막는다.)
- **'문자적 존재'의 도메인 의도:** 정량 수치는 동일한 숫자값이면 표기 차이(예: "40%"와 "40 퍼센트")가 있어도 근거가 있는 것으로 본다. 고유명사 표기 변형의 동일성 판단 등 세부 정규화 규칙은 구현 노트로 분리한다.
- **'성과 주장' 같은 자유 서술은 자동 실패 판정 범위에서 제외한다.** 결정적 추출이 불가능하기 때문이다. 대신 생성 지시(날조 금지)와 사용자 검토로 다룬다. 날조 자체는 정책상 여전히 금지이며, 다만 자동 판정의 대상이 아닐 뿐이다.
- **직접 편집에는 자동 검증을 적용하지 않는다.** 자동 신뢰성 검증은 AI 생성·재생성 결과에만 적용한다. 사용자가 직접 편집으로 넣은 내용의 최종 책임은 사용자에게 있다(최종 결정권은 사용자).
- **검증 실패 시 흐름:** 어떤 생성 항목에서 근거 없는 수치·고유명사가 검출되면 그 항목을 검증실패 상태로 두고 자동으로 1회 재생성한다. 재생성 결과도 검증을 통과하지 못하면 그 항목만 사용자에게 알리고 재시도·직접 편집 경로를 준다(부분 실패와 동일한 회복). 검증실패 항목이 포함된 산출물도 성공 항목을 잃지 않도록 버전으로 보존된다.

### 수용 기준
- 산출물에 등장하는 정량 수치·고유명사 각각은, 그 항목의 출처 경험 기록 안에 문자적 근거가 존재한다(고정밀 사실 근거로 1:1 추적 가능, 근거 없는 항목 0건).
- 사용자 기록에 없는 수치나 고유명사가 검출되면, 해당 항목은 검증실패로 처리되고 자동 재생성 → (재실패 시) 사용자 안내 순으로 회복 경로가 동작한다.
- 각 생성 항목은 자신의 출처 경험 기록 목록(항목 출처 연결)을 가진다(근거 없이 만들어진 항목 0건).
- 경험이 부족한 목표에 대해, 날조 대신 보강 안내가 제시된다.

# MVP 경계와 로드맵

## MVP에 포함 (이번 단계에서 구현)
- 계정 기반 가입·로그인, 사용자별 데이터 영구 보존·삭제.
- 경험 기록 작성·보강(필수: 제목·유형·본문 / 선택: STAR·기간·역량).
- **정적 회상 보조:** 본문 칸 예시 플레이스홀더 + 유도 질문 세트 (AI 호출 없음).
- 목표 정보 직접 입력(채용 방향 텍스트 붙여넣기).
- **이력서 양식 지정:** 프리셋 선택 + 회사 양식 붙여넣기(AI 해석 + 사용자 확정 게이트) + 양식 저장·재사용, 미지정 시 AI 생성 양식(산출물 내 스냅샷). 양식은 이력서 한정(포트폴리오 제외).
- 이력서 생성(전체 1차 생성 → 양식 섹션별 항목 구성), 포트폴리오 생성(경험별 항목, 텍스트 구조물).
- 항목 단위 재생성 + 직접 편집, 버전 보존·비교.
- 결과물 열람 + 텍스트 복사.
- 비용·신뢰성 가드레일.

## 로드맵 (MVP 이후)
- 채용공고 자동 연동(크롤링·연동)으로 목표 정보 입력 간소화.
- 포트폴리오 시각 디자인(레이아웃·테마·이미지)과 PDF/문서 파일 내보내기.
- 경험 기록 작성 보조(AI가 질문을 던져 경험을 끌어내는 인터뷰형 보강 — 정적 회상 보조의 심화 버전).
- 목표 정보 기반 양식 자동 추천(목표를 보고 적합한 이력서 양식을 자동 제안 — 수동 선택과 구별되는 별개 가치). 다국어 산출물.
- 이력서 양식 자유 폼빌더(섹션 직접 추가·삭제·정렬), AI 생성 양식의 재사용 양식 라이브러리화, 포트폴리오 양식.
- 모바일 클라이언트.

> 경계 근거: MVP는 핵심 가치 흐름(기록→겨냥→생성→다듬기→활용→재사용)을 텍스트 수준에서 완결하는 데 집중한다. 외부 연동·시각화·파일 서식은 본질 가치가 검증된 뒤 더한다.

# 측정 가능한 수용 기준 (전체 요약)
개발자가 명세만으로 pass/fail을 판단할 수 있도록, 각 기능의 핵심 기준을 모은다.

1. 경험 기록: 제목·유형·본문 중 하나라도 비면 저장 거부, 누락 항목이 사용자에게 표시된다.
2. 경험 기록: 저장된 경험은 이후 생성 후보 재료에 100% 등장한다.
3. 경험 기록: 본문 칸이 비어 있을 때 예시 플레이스홀더와 유도 질문 세트가 표시된다(정적 회상 보조).
4. 목표 정보: 채용 방향 텍스트가 비면 생성이 거부되고 이유가 안내된다.
5. 목표 정보: 같은 경험 묶음에 다른 목표 정보를 주면 결과 강조점이 달라진다(사람이 비교 식별 가능).
6. 이력서·포트폴리오: 모든 사실 진술이 경험 묶음에 근거를 가진다(근거 없는 사실 0건).
7. 이력서·포트폴리오: 생성 결과가 즉시 하나의 버전으로 저장된다.
8. 생성: 빈 경험 묶음으로는 산출물이 생성되지 않는다.
9. 생성: 부분 생성 실패 시 성공 항목은 유지되고 실패 항목만 재시도 경로가 제공된다.
10. 재생성: 한 항목 재생성이 다른 항목을 바꾸지 않는다.
11. 재생성·편집: 직전 버전이 보존되어 비교·복원이 가능하다(보관 상한 이내).
12. 활용: 저장된 산출물은 이후 언제든 열람되고, 전체·항목 텍스트를 복사할 수 있다.
13. 데이터 소유: 한 사용자의 데이터는 다른 사용자에게 노출되지 않는다.
14. 데이터 소유: 재로그인 후 기존 데이터가 보존되고, 계정 삭제 후엔 모두 조회되지 않는다.
15. 가드레일: 각 상한 도달 시 작업이 차단되고 대안이 안내된다.
16. 가드레일: 경험 근거 없는 사실은 어떤 경로(생성·재생성·개선 지시)로도 추가되지 않는다.
17. 신뢰성 가드레일: 산출물의 정량 수치·고유명사 각각은 출처 경험 기록 내 문자적 근거로 1:1 추적된다(결정적 대조, 근거 없는 항목 0건). '성과 주장' 등 자유 서술은 자동 판정 대상이 아니라 생성 지시·사용자 검토로 다룬다.
18. 신뢰성 가드레일: 근거 없는 수치·고유명사가 검출된 항목은 검증실패로 처리되어 자동 1회 재생성 후 재실패 시 사용자 안내·재시도 경로가 제공된다.
19. 버전: 버전은 산출물 단위 스냅샷이며 산출물은 하나의 활성 버전을 가리킨다. 항목 채택 시 변경 항목만 교체한 새 버전이 활성이 되고 나머지 항목은 그대로 복제된다.
20. 동시성: 한 생성 항목은 동시에 하나의 진행 중 생성만 가지며, 같은 항목의 중복 재생성 요청은 거절된다.
21. 양식 지정: 사용자가 지정한 양식의 각 필수 섹션 정의에, 근거 경험이 있는 한, 대응하는 생성 항목이 산출물에 1:1로 존재한다(결정적 대조).
22. 양식 스냅샷: 한 산출물의 모든 버전은 동일한 양식 스냅샷(섹션 정의 집합과 순서)을 참조하며, 버전 간 섹션 정의 집합의 차이가 0이다. 양식 미지정 시 AI 생성 양식이 스냅샷(섹션 정의 집합과 순서)으로 보존되어, 이후 열람 시 동일한 섹션 정의 집합과 순서가 조회된다.
23. 양식과 가드레일: 양식이 요구한 섹션에 근거 경험이 0이면 그 섹션은 생성 항목으로 만들어지지 않는다. 필수 섹션은 부족이 고지되고(생성은 진행) 선택 섹션은 생략되며, 어느 경우에도 날조로 채워지지 않는다(근거 없는 항목 0건 유지).
24. 양식 해석 게이트: 붙여넣기로 만든 양식은 AI 해석 결과를 사용자가 확정한 뒤에만 생성에 적용된다.

# 명세서의 규칙
- 오직 **도메인 관점**(무엇을, 왜)에 집중한다.
    - *구현 방법*에 관한 세부(직렬화 포맷·저장소·삭제 전략 등)는 도메인 서술에 섞지 않고, 구현 노트 문서에 별도로 모은다.
    - 목적: 도메인 규칙이 바뀔 때와 구현이 바뀔 때를 분리해 추적할 수 있게 만든다.
- 각 기능에 대한 명세는, 도메인 관점(무엇을, 왜)에서 다음 항목들을 충분히 구체적으로 설명해야 한다.
    - 이 기능의 목적
    - 이 기능을 통해 이뤄낼 수 있는 가치
    - 사용자가 이 기능을 사용할 때의 구체적인 시나리오(동작 방식, 성공 케이스, 실패 케이스 등)
- 하나의 도메인에 대해 여러 용어를 혼용해 사용하지 않는다.
    - 도메인의 영어 명칭은 제목에서 한 번 드러낸 후, 아래에선 한글 명칭만 표현한다.
    - 해야 하는 것의 예시: 도메인 용어 정리 항목에서 '이력서(Resume)'으로 표현한 후, 이후 서술에서는 '이력서'라고만 명시
    - 하지 말아야 하는 것의 예시: '이력서'이라는 도메인에 대해, '지원 서류', 'CV' 등 여러 표현 사용
