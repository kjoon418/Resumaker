이 문서는 트랜잭션 분리 지침을 위해 작성되었다.

# 트랜잭션 분리의 기준

어떤 동작을 하나의 트랜잭션으로 관리할지, 여러 트랜잭션으로 분리할지는, 다음 요소들을 중심으로 트레이드 오프를 고려해 결정한다.
하나의 요소만을 보지 않고, 모든 요소를 종합적으로 고려한다.

단, '데이터 정합성 관점'을 최우선으로 고려한다.
만일 방어 로직을 구성하더라도 데이터 정합성 훼손이 문제가 될 동작은, 반드시 하나의 트랜잭션으로 관리한다.
(방어 로직, 스케줄러, 이벤트 핸들러 구현 등으로 대처할 수 있는 데이터 정합성 훼손 동작은, 다른 요소를 같이 고려해 결정한다)

---

# 트랜잭션 분리를 고려할 때 사용할 관점

## 데이터 정합성 관점

다음 동작들은 하나의 트랜잭션으로 합치는게 유리하다:
- 일부만 성공했을 때 데이터 정합성이 깨지는 동작들
- 데이터 정합성이 깨지면 즉시 문제로 이어지는 동작들
    - 예시: 계좌이체의 경우, 인출은 성공하고 입금은 실패할 경우 즉시 문제가 된다.

다음 동작들은 별개의 트랜잭션으로 분리하는게 유리하다:
- 일부만 성공하더라도 데이터 정합성이 깨지지 않는 동작들
- 데이터 정합성이 깨지더라도 후속 조치로 해결할 수 있는 동작들
    - 예시: '알림 전송'은 실패하더라도 추후에 재시도하면 된다.

## 동작의 개념 관점

다음 동작들은 하나의 트랜잭션으로 합치는게 유리하다:
- 실제론 여러 동작으로 구성되어 있더라도, 개념상 하나의 동작인 동작들
    - 예시: 수정 기능이 내부적으로 '삭제 후 생성'으로 구현했다면, '삭제 후 생성'은 개념상 하나의 동작이다.

다음 동작들은 별개의 트랜잭션으로 분리하는게 유리하다:
- 개념상 메인 동작과 부가 동작으로 구분할 수 있는 동작들
    - 예시: '회원가입 후 이메일 전송' 기능은 메인 동작(회원가입)과 부가 동작(이메일 전송)으로 구분할 수 있다.

## 동작의 실패 확률 관점

다음 동작들은 하나의 트랜잭션으로 합치는게 유리하다:
- 실패 확률이 매우 낮은 부가 동작
    - 예시: '회원탈퇴 후 히스토리 생성' 기능의 '히스토리 생성' 동작은 단순히 레코드를 삽입할 뿐이니 실패 확률이 낮다.

다음 동작들은 별개의 트랜잭션으로 분리하는게 유리하다:
- 실패 확률이 낮지 않은 부가 동작
    - 예시: '이메일 발송', '외부 API', '동시성 위험이 있는 DB 조작' 등

## 유지보수 관점

다음 동작들은 하나의 트랜잭션으로 합치는게 유리하다:
- 여러 트랜잭션으로 분리하려면 구현이 많이 복잡해지는 동작들
- 하나의 서비스에서 모두 처리하는게 직관적이고 가독성에 유리한 동작들

다음 동작들은 별개의 트랜잭션으로 분리하는게 유리하다:
- 여러 트랜잭션으로 분리하더라도 간단히 구현할 수 있는 동작들
- 하나의 서비스에서 모두 처리하기 부담되는 동작들(가독성, 유지보수 측면)

## 트랜잭션 비용 관점

다음 동작들은 하나의 트랜잭션으로 합치는게 유리하다:
- 한 트랜잭션으로 묶더라도 수행 시간이 문제되지 않는 동작들

다음 동작들은 별개의 트랜잭션으로 분리하는게 유리하다:
- 한 트랜잭션으로 묶으면 '롱 트랜잭션'이 될 우려가 있는 동작들

# 트랜잭션 분리 방법

트랜잭션을 분리할 때 우선적으로 고려할 수 있는 방법들이다.
반드시 이 방법만을 사용할 필요는 없으며, 현재 상황을 기준으로 트레이드 오프를 고려해 최선의 도구를 사용한다.

단, 아래 도구를 제외한 다른 도구를 선택했을 경우, 해당 도구를 선택한 근거를 반드시 제시해야 한다.

## 이벤트 핸들러

서비스에서 `ApplicationEventPublisher`를 이용해 이벤트를 던지고, 이벤트 핸들러가 별도 트랜잭션에서 기능하도록 구현한다.

부가 동작 처리를 위한 '이벤트', '이벤트 핸들러', '부가 동작 객체' 등은 하나의 패키지에서 관리한다.
이 때, 패키지 이름으로 부가 동작을 표현한다.

구체적인 예시는 다음과 같다.

```java
package myapplication.feature.user.notice;

// ...

public class SignUpCompleteHandler {

    private final SignUpNotificater notificater;

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void noticeSignUpCompleted(SignUpCompleteEvent event) {
        notificater.notice(event.userId());
    }
}
```

## 스케줄러

동시성 문제 등으로 인해 이벤트 핸들러가 실패할 확률이 존재할 경우, 안전장치로써 별도의 스케줄러를 구현한다.

이벤트 핸들러와 스케줄러가 중복 로직을 최소한으로만 갖도록, 부가 동작 자체는 별도 객체로 분리할 것을 권장한다.

구체적인 예시는 다음과 같다.

```java
package myapplication.feature.user.notice;

// ...

public class SignUpNoticeScheduler {

    private final UserRepository repository;
    private final SignUpNotificater notificater;

    @Scheduled(cron = "0 0 * * * *")
    @SchedulerLock(name = "signUpNotice", lockAtLeastFor = "PT30S", lockAtMostFor = "PT10M")
    public void noticeNotSentSignUpCompleted() {
        List<Long> userIds = repository.findSignUpNoticeNotSentUserIds();
        
        for (Long userId : userIds) {
            notificater.notice(userId);
        }
    }
}
```
