# 광고 — Google AdSense 도입 전략

> 목표: "산출물 생성 대기 시간을 광고로 수익화"한다는 기획(`docs/광고-도입-기획.md`)을 **Google AdSense**로 실현하기 위한 전략. 자사 플레이스홀더 단계(`docs/광고-운영-핸드오프.md` §3 "실제 광고망 연동")의 구체화 버전이다.
>
> 결론을 먼저 말한다. **현재 구조에 AdSense를 그대로 붙이면 거의 확실히 거절·정책 위반된다.** 붙이는 일이 아니라 *승인받을 수 있는 표면을 먼저 만드는* 일이다. 아래는 그 현실적 경로다.

---

## 0. 한 문단 결론 (TL;DR)

AdSense는 **크롤 가능한 텍스트 콘텐츠가 있는 공개 페이지**에 광고를 싣는 제품이다. 이 앱은 (1) 전 화면을 `<canvas>` 픽셀로 그리는 Compose Wasm이라 DOM에 읽을 텍스트가 없고, (2) 핵심 화면이 로그인 뒤에 있다. 둘 다 AdSense 승인의 정면 위배다. 따라서 전략의 핵심은 **"대기 화면에 AdSense를 욱여넣는 것"이 아니라, AdSense가 승인·서빙할 수 있는 공개 콘텐츠 표면(랜딩 + 이력서 작성 가이드/블로그 등 실제 HTML)을 만들어 거기서 수익을 내고, 인앱 대기 화면은 자사 플레이스홀더를 유지(또는 AdSense가 아닌 더 적합한 수단을 별도 검토)**하는 것이다.

---

## 1. 왜 "그냥 못 붙이나" — 세 개의 벽

### 벽 1. 승인·크롤 (가장 큰 차단)
- AdSense 가입 심사와 상시 정책 크롤러는 페이지의 **DOM 텍스트 콘텐츠**를 평가한다.
- 이 앱 `app/webApp/src/webMain/resources/index.html`의 `<body>`는 **로딩 스피너 SVG 하나**뿐이고, 실제 화면은 전부 Compose가 `<canvas>`에 **픽셀로** 그린다. 크롤러에게는 **콘텐츠가 없는 빈 페이지**로 보인다 → 대표 거절 사유 "가치가 낮은 콘텐츠 / 콘텐츠 없음".
- 더구나 산출물 대기 화면을 포함한 핵심 흐름은 **로그인 뒤**에 있어 심사자/크롤러가 **도달 자체를 못 한다**.

### 벽 2. 렌더 (기술적 불일치)
- AdSense는 DOM 요소 `<ins class="adsbygoogle">`에 `adsbygoogle.js`가 광고를 주입하는 구조다. **캔버스 안에는 DOM을 넣을 수 없다.**
- 캔버스 위에 HTML 오버레이 `div`를 띄워 Compose 슬롯 좌표에 동기화해야 하는데, 스크롤·리사이즈·폴링 중 슬롯의 등장/소멸과 위치를 맞추는 동기화가 **취약**하고, AdSense 정책상 **콘텐츠를 가리는 오버레이·플로팅 광고는 제한**된다.

### 벽 3. 정책 적합성 (대기 화면 자체가 위험)
- AdSense 정책: **콘텐츠가 거의 없는 페이지**, **이용 중단/오류/로그인/전환 같은 기능성 화면**에는 광고를 두지 못한다. 산출물 대기 화면은 정확히 이 "콘텐츠 빈약·과도기 기능 화면"에 해당한다.
- **EEA/영국 사용자**에게 광고를 서빙하려면 Google 인증 **CMP(동의 관리 플랫폼)** 가 의무다(미설치 시 서빙 차단).
- 개인정보처리방침 필수, 클릭 유도/우발 클릭 유발 배치 금지, "광고"임을 구분 가능하게 표기.

> 요약: 벽 1(승인)이 모든 것의 선결이고, 그것이 **공개 크롤 가능 콘텐츠 표면**을 요구한다. 벽 2·3은 "대기 화면 인앱 AdSense"를 사실상 비권장으로 만든다.

## 2. 전략 개요 — 어디에 광고를 싣는가 (핵심 의사결정)

| 옵션 | 내용 | 장점 | 단점 | 판정 |
| --- | --- | --- | --- | --- |
| **A. 공개 콘텐츠 표면에 AdSense (권장)** | 크롤 가능한 실 HTML 페이지(랜딩 + 이력서 작성 가이드/블로그/도움말)를 만들고 거기에 AdSense를 싣는다. 인앱 대기 화면은 자사 플레이스홀더 유지. | 승인·정책·기술 3중 벽을 모두 정면 통과. 캔버스 제약과 무관. SEO 유입까지 덤. | 콘텐츠(글) 제작·운영 비용. 수익이 "대기 시간"이 아니라 "콘텐츠 트래픽"에서 남. | **채택** |
| B. 대기 화면에 AdSense를 오버레이로 강제 | 캔버스 위 HTML 오버레이로 adsbygoogle 주입. | 원래 의도(대기 시간 수익화)에 직접 부합. | 승인 난망(콘텐츠 없는 페이지)+정책 위반(기능 화면)+동기화 취약+계정 정지 리스크. | 비권장 |
| C. AdSense 포기, 인앱은 다른 수단 | 대기 화면은 자사 홍보/제휴/직접판매 배너 유지, 외부망은 추후. | 정책 충돌 0, 현재 코드 그대로. | 즉각적 외부 광고 수익 없음. | 보조(A와 병행 가능) |

**근거:** 사용자의 원래 목표는 "토큰 비용 충당을 위한 광고 수익"이다. 그런데 AdSense라는 **수단**의 구조적 제약(크롤 가능 콘텐츠 필요)이 "대기 화면"이라는 **위치**와 충돌한다. 목표(수익)는 위치에 종속되지 않으므로, **AdSense를 쓰되 위치를 "승인 가능한 공개 콘텐츠 표면"으로 옮기는** A가 유일하게 승인까지 도달하는 경로다. 대기 화면 수익화는 AdSense가 아니라 자사/제휴(C)로 분리한다.

> ⚠️ **이 결정이 후속 작업량을 크게 가른다.** A를 택하면 "콘텐츠 표면 제작"이 새 워크스트림으로 추가된다. 진행 전 이 방향(A 채택, 대기 화면은 자사 유지)에 동의가 필요하면 알려달라 — 동의 시 Phase 1부터 착수한다.

## 4. 이 프로젝트에 맞춘 기술 통합 설계

> 기존 런타임 구성 패턴(`config.js` + nginx `40-api-base.sh` + `window.__RESUMAKER_*__`)을 그대로 재사용하면 **이미지 재빌드 없이** 퍼블리셔 ID·on/off를 환경마다 주입할 수 있다.

### 4.1 퍼블리셔 ID 런타임 주입 (재빌드 없이) — ✅ 구현됨
**환경변수 `ADSENSE_CLIENT`(`ca-pub-…`) 하나**로 ads.txt·소유권 확인 메타 태그·config.js 전역이 컨테이너 기동 시 자동 생성된다. 빈 값이면 전부 무동작(graceful).
- **전달 경로(중요):** `.env(.prod)` → `compose.yaml`의 `frontend.environment: ADSENSE_CLIENT: ${ADSENSE_CLIENT:-}` → 컨테이너 env → `40-api-base.sh`가 굽는다. **`compose.yaml`에 패스스루가 없으면 컨테이너까지 닿지 않는다**(같은 이유로 기존 `ADS_ENABLED`도 이번에 패스스루를 추가해 실제로 동작하게 고침).
- `config.js`/`.env.example`/`.env.prod.example`에 빈 기본값과 설명을 둔다. 운영은 `.env.prod`(커밋 금지)에 실제 ID를 넣고 재기동만 하면 된다(재빌드 불필요).

### 4.2 소유권 확인 (ads.txt + 메타 태그) — ✅ 구현됨
`ADSENSE_CLIENT`가 있으면 `40-api-base.sh`가 둘을 동시에 생성한다(둘 병행 시 확인이 더 빠르고 안정적):
- **ads.txt**: 웹 루트 `/ads.txt`에 `google.com, pub-…, DIRECT, f08c47fec0942fa0`(ca- 접두 제거). `default.conf`의 `location = /ads.txt`가 text/plain으로 서빙(SPA 폴백 제외).
- **메타 태그**: `index.html` `<head>`의 토큰 `<!--__ADSENSE_META__-->`을 `<meta name="google-adsense-account" content="ca-pub-…">`로 치환. **정적 HTML에 박혀** 크롤러가 JS 실행 없이 읽는다.
- ⚠️ **소유권 확인 ≠ 승인**: 위 둘은 "도메인이 내 것"만 통과시킨다. 콘텐츠 심사(승인)는 여전히 **공개 크롤 가능 콘텐츠 표면(Phase 1)** 이 있어야 통과한다.

#### ⚠️ ads.txt와 "루트 도메인 → app 서브도메인" 리다이렉트 (이 프로젝트 배포 전제)
AdSense는 **루트 도메인**(`<도메인>`)만 등록되고, Cloudflare가 루트를 `app.<도메인>`(프런트가 실제로 뜨는 호스트)으로 리다이렉트하는 구성이다. ads.txt 크롤러는 루트 `<도메인>/ads.txt`를 보는데, `app.<도메인>`은 **같은 루트 도메인(eTLD+1)** 이라 IAB ads.txt 스펙상 **내부 리다이렉트로 따라가진다** → `<도메인>/ads.txt`→`app.<도메인>/ads.txt`(컨테이너가 서빙)는 인정된다. 단 **필수 조건**:
- **경로 보존(path-preserving) 리다이렉트**여야 한다. `<도메인>/ads.txt`가 `app.<도메인>/ads.txt`로 가야지, app 홈(`app.<도메인>/`)으로 통째 보내면 ads.txt·콘텐츠(`/guide`)가 전부 깨진다. Cloudflare Redirect Rule에서 **Preserve query string/path** 켜기.
- **301 한 홉**으로(루프 금지: app이 다시 루트로 되돌리지 않기). `http→https→app` 2홉 이상 지양.
- `SITE_ORIGIN`은 **리다이렉트 도착지인 `https://app.<도메인>`** 으로 둔다(canonical은 200으로 응답하는 최종 URL을 가리켜야 함 — 루트로 두지 말 것).
- 배포 후 검증: `curl -sI https://<도메인>/ads.txt`의 `Location`이 `…app.<도메인>/ads.txt`(끝에 `/ads.txt` 유지)면 정상, `…app.<도메인>/`(루트)면 경로 보존 꺼진 것. `curl -sL https://<도메인>/ads.txt`로 실제 ads.txt 내용 확인, `curl -sI https://<도메인>/guide`도 `/guide` 유지 확인.

### 4.3 AdSense 로더(광고 유닛) 주입 위치 — ⏳ ID 발급·승인 후
- **공개 콘텐츠 표면(권장)**: 그 표면의 `<head>`에 `adsbygoogle.js` 로더 + 유닛. **앱(index.html)에는 넣지 않는다**(캔버스엔 효과 없음·정책 위험). 소유권 확인용 메타 태그(§4.2)와 달리, 실제 광고 로더는 콘텐츠 표면에만 둔다.
- 만약 Phase 4까지 가서 인앱 오버레이를 시도한다면: `index.html`에 로더를 두되 `__RESUMAKER_ADSENSE_CLIENT__`가 있을 때만, 캔버스 위 오버레이 `div`를 Compose 슬롯 좌표에 동기화(별도 위험 평가 필수).

### 4.4 기존 코드 자산 재사용
- 인앱 슬롯을 끝내 외부 콘텐츠로 바꾸는 날에도 **`AdContentProvider` 어댑터 교체**만으로 끝난다(슬롯·게이팅·"광고" 라벨·정당화 카피·`adsEnabled()` 플래그·`AdMetricsReporter` 전부 재사용). `docs/광고-운영-핸드오프.md` §5 참고.

## 5. 정책·법무 체크리스트 (Phase 1~2 동반)

- [ ] **개인정보처리방침** — 제3자(Google) 광고·쿠키·데이터 사용 고지.
- [ ] **CMP(동의)** — EEA/영국 트래픽이 있으면 Google 인증 CMP 의무.
- [ ] **`ads.txt`** — 퍼블리셔 ID 등록(무효 트래픽·도용 방지).
- [ ] **"광고" 표기** — 한국 인터넷 광고 규정상 광고임을 명확히. AdSense 본문 광고도 라벨 유지.
- [ ] **클릭 유도 금지** — "광고를 클릭하세요" 류 문구·우발 클릭 유발 배치 금지(계정 정지 사유).
- [ ] **무효 클릭 보호** — 본인/지인 클릭 금지. 대기 화면 강제 노출형은 무효 트래픽 의심받기 쉬움(Phase 4 비권장 근거 보강).
- [ ] **콘텐츠 정책** — 사용자 업로드 이력서 텍스트가 노출되는 페이지에 광고를 두지 않기(민감 PII).

## 6. 리스크 & 대안

| 리스크 | 영향 | 완화 |
| --- | --- | --- |
| 캔버스 SPA라 심사 거절 | AdSense 진입 자체 불가 | **Phase 1 공개 콘텐츠 표면**으로 우회(이 전략의 핵심) |
| 대기 화면 인앱 강제 노출 → 정책 위반/계정 정지 | 계정 영구 정지 가능 | 대기 화면은 자사/제휴로, AdSense는 콘텐츠 표면에 한정 |
| 콘텐츠 표면 트래픽 미미 → 수익 미미 | 기대 수익 미달 | SEO 가이드 콘텐츠 축적, 또는 대안 네트워크 |
| 운영 비HTTPS·도메인 미정 | 심사 불가 | 운영 HTTPS·도메인 확정 선행(쿠키 인증 메모리와 동일 선결) |

**대안 광고/수익 수단(AdSense 부적합 시):** 자사 제휴(채용 플랫폼·교육 콘텐츠 affiliate)를 대기 화면 슬롯에 직접 판매형으로 — 이는 캔버스 안 순수 Compose로 이미 렌더 가능하고 정책 충돌이 없다(현재 플레이스홀더의 자연스러운 발전형).

**✅ 코드로 끝난 것(승인 전이라도 안전, 빈 값이면 무동작):** `ADSENSE_CLIENT` 환경변수 런타임 주입(`compose.yaml` 패스스루 + `.env(.prod).example` 문서화), `ads.txt` 자동 생성·nginx 서빙(§4.2), 소유권 확인 **메타 태그** 자동 주입(§4.2), `config.js` 전역. 기존 `ADS_ENABLED` 킬스위치의 compose 패스스루 누락도 함께 수정.
