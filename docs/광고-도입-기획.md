# 광고 도입 기획 (대기 시간 광고 슬롯)

> 본 문서는 산출물 생성 대기 시간에 광고를 노출해 LLM 토큰 비용을 충당하기 위한 기능의 **기획 명세**다.
> 기획자(PM)가 섹션 1~5(배경·정당화 전략·노출 정책·요구사항·범위)를 작성했고,
> 디자이너가 이어서 UX 플로우·레이아웃·카피·수용기준을 덧붙인다.
> 작성 원칙: 트레이드오프가 갈리는 지점은 선택지를 나열하고 근거로 하나를 고른다(CLAUDE.md 규칙).
> 추측이 필요한 제품 결정은 합리적 기본값을 정하되 "**결정/가정**"으로 명시한다.

---

## 1. 배경과 목표

### 1.1 왜 광고인가 (문제 정의)

- Resumaker는 사용자의 경험·경력을 LLM으로 가공해 이력서·포트폴리오를 생성한다. **생성 1건마다 토큰 비용이 발생**하며, 사용량이 늘수록 비용이 선형으로 증가한다. 무료로 운영하면 비용 구조가 붕괴한다.
- 가장 직접적인 해법인 **유료 결제는 현재 법적 검토 이슈로 도입할 수 없다.** 따라서 단기적으로 비용을 충당할 다른 수익 수단이 필요하다.
- 한편 생성은 **비동기**이며 사용자에게는 명백한 "대기 시간"이 존재한다. 이 대기 시간은 현재 **순수 비용**(사용자는 기다리기만 하고, 우리는 토큰만 태운다)이다. 이 구간을 **광고 노출 기회**로 전환하면, 비용을 만든 바로 그 행위(생성)의 대기 시간에서 비용을 회수하는 구조가 된다.

### 1.2 확정된 구현 방향 (사용자 승인)

- **"교체 가능한 광고 슬롯 인프라 + 자체 플레이스홀더"**로 시작한다.
- 지금 단계는 실제 광고망(AdSense 등) 연동이 **아니다.** 교체 가능한 광고 슬롯 컴포넌트를 만들고, 그 안에 **자사 안내·제휴 배너 같은 플레이스홀더**를 채운다.
- 계정·법적 검토가 끝나면 같은 슬롯에 **실제 광고만 끼워 넣는다.** 즉 이번 작업의 핵심 산출물은 "광고가 들어갈 자리(seam)"와 "그 자리를 사용자에게 정당하게 제시하는 방식"이다.

### 1.3 성공 정의

이 기능의 성공은 **두 축의 균형**으로 정의한다. 한쪽만 달성하면 실패다.

- **(A) 비용 충당 잠재력 확보** — 실제 광고망이 끼워지면 즉시 수익이 발생할 수 있는 인프라·노출량·계측이 갖춰진다.
- **(B) UX 저하 최소화** — 광고 도입 전후로 핵심 지표(생성 완주율, 이탈률)가 유의미하게 나빠지지 않는다.

### 1.4 측정 가능한 목표 (지표)

> 절대 수치 목표는 트래픽·실광고 단가가 없는 현 단계에서 신뢰도가 낮으므로, **계측 가능성과 상대 변화**를 1차 목표로 둔다. (결정 근거는 5.4 리스크 참조)

| 구분 | 지표 | 목표 |
| --- | --- | --- |
| 인프라 (A) | 광고 슬롯 노출 이벤트(impression) 계측 커버리지 | 대기 카드 노출의 **100%**가 이벤트로 잡힌다 |
| 인프라 (A) | 실광고 교체 작업량 | 슬롯에 실제 광고 어댑터를 끼울 때 **기획·UX 변경 0건**(콘텐츠 소스만 교체) |
| UX (B) | 생성 완주율(제출 → 완성 열람) | 광고 도입 전 대비 **하락 없음**(비열등성, 허용 마진은 디자이너/데이터와 확정) |
| UX (B) | 대기 화면 이탈률 | 광고 도입 전 대비 **악화 없음** |
| UX (B) | 광고 영역 명시 라벨 인지 | 노출되는 모든 광고 슬롯에 "광고/Sponsored" 라벨 **100%** 표기 |

- **결정:** 1차 출시의 핵심 KPI는 "노출 계측 커버리지 100%"와 "완주율 비열등성"이다. 광고 클릭률·eCPM 같은 수익 지표는 실광고 연동 이후의 2차 KPI로 미룬다(현재는 의미 있는 값이 나오지 않음).

## 2. 광고 이론 기반 정당화 전략

핵심 프레이밍: **광고를 "강제 대기의 생산적 전환" + "가치 교환의 투명한 제시"로 제시한다.** 사용자가 "광고를 당했다"가 아니라 "이게 왜 여기 있는지 이해되고, 받아들일 만하다"고 느끼게 만드는 것이 최우선 요건이다. 아래 이론들을 실제로 적용한다.

### 2.1 가치 교환 / 상호성 (Value Exchange / Reciprocity)

- **이론:** 사람은 무언가를 받으면 갚으려는 경향이 있다(상호성). 광고에서는 사용자가 명시적 비용 없이 가치(무료 생성)를 받는 대신, 주의(attention)를 제공하는 거래가 성립할 때 광고 수용도가 높아진다. 특히 **거래가 투명하게 명시될 때** 반발이 줄어든다.
- **Resumaker 적용:** 대기 영역에 "AI 생성은 비용이 듭니다. 광고가 이 서비스를 무료로 유지합니다" 같은 **교환의 근거를 1줄로 투명하게** 노출한다. 광고를 몰래 끼우지 않고, 왜 보게 되는지를 먼저 말한다.
- **사용자가 받는 메시지:** "내가 공짜로 쓰는 대가가 이거구나. 합리적이다."

### 2.2 맥락 적합성 (Contextual Relevance / Ad Context Congruence)

- **이론:** 광고는 사용자의 현재 맥락·과업과 어긋날수록 침입(intrusion)으로 인식되고, 맞물릴수록 자연스럽게 수용된다. 과업 흐름을 끊는 위치(예: 핵심 입력 도중)의 광고가 가장 거슬린다.
- **Resumaker 적용:** 광고를 **생성이라는 비용 발생 행위의 대기 구간에만** 둔다. 입력·편집·열람처럼 사용자가 집중·작업 중인 화면에는 두지 않는다. 광고의 위치 자체가 "이 비용을 만든 작업을 기다리는 동안"이라는 맥락을 갖는다.
- **사용자가 받는 메시지:** "내가 시킨 작업이 돌아가는 동안 잠깐 보는 거구나. 흐름을 방해받지 않았다."

### 2.3 강제 대기의 생산적 전환 (Productive Reframing of Forced Wait / Operational Transparency)

- **이론:** 비어 있는 대기(빈 스피너)는 체감 시간이 길고 불안을 키운다. 같은 대기라도 **무언가 일어나고 있음을 보여주면** 체감 시간이 줄고 만족이 는다(노동 환상, operational transparency). 또한 어차피 기다려야 하는 시간에 콘텐츠가 채워지면 대기 자체의 비용 인식이 낮아진다.
- **Resumaker 적용:** 대기 영역을 **"진행 상태(생성 중/대기 중) + 광고"가 공존하는 생산적 공간**으로 설계한다. 광고는 "빈 대기를 채우는 콘텐츠"로 위치하지, "진행을 가로막는 관문"으로 위치하지 않는다. (진행 표시를 광고가 가리거나 대체하지 않는다 — 3.5 엣지 케이스.)
- **사용자가 받는 메시지:** "그냥 멍하니 기다리는 게 아니라 뭔가 보면서 기다리는구나. 생성도 진행 중인 게 보인다."

### 2.4 보상형 광고 멘탈모델 (Rewarded-Ad Mental Model)

- **이론:** 보상형 광고(게임 등)는 "광고를 보면 보상을 받는다"는 명시적 거래라 수용도가 매우 높다. 사용자가 **자발적·기대된 교환**으로 인식하기 때문이다.
- **Resumaker 적용:** 여기서 "보상"은 곧 **완성된 산출물**이다. 광고와 완성을 같은 대기 타임라인 위에 놓아 "이 광고를 보는 시간 끝에 내 이력서가 나온다"는 멘탈모델을 자연스럽게 형성한다. 단, **보상을 인질로 잡지 않는다**(광고 시청을 완성의 전제 조건으로 강제하지 않는다 — 보상형의 강압 버전은 다크패턴이다). 생성은 광고와 무관하게 진행되고 완료되며, 광고는 그 위에 얹힌 부가 레이어다.
- **사용자가 받는 메시지:** "기다리면 결과물이 나오고, 그 시간에 광고가 같이 있는 거구나. 광고 때문에 내 결과가 늦어지는 건 아니다."

### 2.5 투명성·통제감 (Transparency & Sense of Control)

- **이론:** 광고가 무엇인지 속이거나(기만), 사용자가 통제할 수 없게 만들면 신뢰가 무너진다. 반대로 **명확한 라벨링**과 **최소한의 통제권**은 수용도와 브랜드 신뢰를 동시에 높인다.
- **Resumaker 적용:** 모든 광고 슬롯에 **"광고/Sponsored" 라벨을 명시**한다(콘텐츠로 위장 금지). 자사 안내·정직한 상태 표시와 시각적으로 구분한다. 통제감은 "광고가 진행을 막지 않고, 완성되면 즉시 사라진다"는 **예측 가능성**으로 우선 제공한다. (광고 숨기기/닫기 같은 명시적 컨트롤 제공 여부는 디자이너 UX 섹션에서 확정 — 4.1 FR-7 참조.)
- **사용자가 받는 메시지:** "이건 광고라고 분명히 적혀 있고, 끝나면 알아서 사라진다. 속았다는 느낌이 없다."

### 2.6 정당화 전략 요약

> 한 줄 원칙: **"비용이 드는 작업을 무료로 제공하기 위해, 그 작업을 기다리는 시간에만, 투명하게 라벨링된 광고를 보여준다."** — 위치(맥락 적합)·시점(대기 한정)·투명성(라벨)·비강압(보상 비인질)이 네 기둥이다. 이 네 기둥 중 하나라도 무너지면 정당화는 깨진다.

## 3. 노출 위치 · 시점 · 빈도 정책

원칙: **최소 침습(minimum intrusion).** 광고는 기존 UX 원칙(특히 "막다른 길 금지", "정직한 안내", "감정적 다크패턴 금지")과 충돌하지 않아야 한다.

### 3.1 노출 위치 (Where)

- **1차 자리 (이번 범위):** 산출물 목록 화면(`ArtifactListScreen`)의 **진행 중 작업 카드 구간**. 구체적으로 `state.renderJobs` 중 **활성(PENDING/RUNNING) 작업이 있을 때**, 진행 카드 영역에 연동된 광고 슬롯을 둔다.
  - 기술 전제: 화면 전체가 단일 캔버스(Compose Wasm/skiko)라 DOM 광고 SDK는 오버레이가 필요하지만, **플레이스홀더 광고는 순수 Compose 컴포저블로 렌더 가능**하다. 따라서 1차 슬롯은 별도 DOM 오버레이 없이 카드 레이아웃 안에 자연스럽게 들어간다.
- **배치 형태 선택지 (트레이드오프 → 결정):**
  - (a) **활성 작업 카드 묶음 아래 단일 슬롯**: 진행 카드 흐름을 끊지 않고 하단에 1개. 침습 최소, 노출은 1개로 제한.
  - (b) 각 활성 작업 카드 사이마다 인터리브: 노출량 ↑, 그러나 진행 카드 가독성·"청킹" 원칙 훼손, 작업 여러 개일 때 광고 도배 위험.
  - (c) 진행 카드 내부 임베드: 진행 상태와 광고가 한 카드에 섞여 정직성·라벨 분리가 어려움.
  - **결정 → (a)**. "단일 목적 화면/청킹" 원칙과 최소 침습에 가장 부합. 활성 작업이 여러 개여도 광고는 **세션당 화면에 1개 슬롯**으로 제한(빈도 정책 3.3과 연동). 노출량은 실광고 연동 시 재검토.
- **2차 자리 (비범위, seam만):** 제작 화면 제출 직후~목록 진입 전환 구간, 또는 단일 산출물 열람 로딩 등은 이번에 만들지 않되, 슬롯 컴포넌트를 재사용 가능하게 설계해 **확장 seam**을 남긴다.

### 3.2 노출 시점 (When)

- 광고 슬롯은 **생성이 실제 진행 중일 때만**(해당 화면에 PENDING/RUNNING 작업이 1개 이상 존재) 노출한다.
- 활성 작업이 없으면(완성만 있거나, 빈 상태) **슬롯을 렌더하지 않는다.** 빈 상태(`EmptyState`)·완성 목록만 있는 화면에는 광고가 끼지 않는다 — 대기 맥락(2.2)이 없는 곳엔 광고도 없다.

### 3.3 노출 빈도 (How often)

- **동시 노출 상한:** 활성 작업 수와 무관하게 **화면당 광고 슬롯 1개**. 작업이 3개 돌아도 광고는 1개.
- **갱신/순환:** 1차는 플레이스홀더이므로 **대기 1회 동안 콘텐츠 고정**(깜빡임·잦은 교체로 인한 주의 분산 방지, "목적 없는 장식 배제" 원칙). 폴링 주기(`POLL_INTERVAL_MS=3000`)에 맞춰 광고를 교체하지 않는다 — 폴링은 상태 갱신용이지 광고 리프레시 트리거가 아니다.
- **세션 빈도:** 별도 일/세션 노출 캡은 1차에서 두지 않는다(노출 자체가 "대기 중"으로 자연 제한됨). 실광고 연동 시 프리퀀시 캡을 도입할 seam만 남긴다.

### 3.4 즉시 철거 규칙 (생성 완료 시)

- 폴링으로 작업이 **SUCCEEDED로 전환되어 화면에서 활성 작업이 모두 사라지면, 광고 슬롯은 즉시 사라진다.** 완성 순간 사용자의 주의는 결과물로 가야 하며, 광고가 보상(결과 열람)을 가로막아선 안 된다(2.4 비인질 원칙).
- "완성됐는데 광고를 더 보라"는 식의 잔류·끼어들기 금지.

### 3.5 엣지 케이스 정책

- **생성 실패(FAILED):** 광고를 **띄우지 않는다.** 실패 카드는 사용자의 공감·복구가 최우선 맥락이며(UX 9번 "엣지 케이스 공감"), 그 위에 광고를 얹는 것은 감정적 다크패턴에 가깝다. 활성 작업이 전부 FAILED면 광고 슬롯 미노출.
- **진행 표시 보호:** 광고가 진행 배지("생성 중/대기 중")·진행 라인(`IndeterminateProgressLine`)·스피너를 **가리거나 대체하지 않는다.** 광고는 진행 표시와 시각적으로 분리된 별도 영역이다.
- **매우 빠른 완료:** 생성이 광고 1회 노출 전에 끝나면, 광고를 억지로 보여주지 않는다(완성 우선). 노출이 0건일 수 있으며 이는 정상이다.
- **다중 활성 작업:** 3.3에 따라 슬롯 1개로 묶어 표시. 작업마다 광고를 늘리지 않는다.
- **오프라인/슬롯 콘텐츠 로드 실패:** 광고 콘텐츠를 못 불러오면 **슬롯을 조용히 접는다**(빈 박스·깨진 영역·에러 노출 금지). 광고 부재가 화면 레이아웃을 깨뜨리지 않는다(graceful degradation).
- **접근성 대형 폰트:** 글자 크기를 최대로 키워도 광고 슬롯이 진행 카드·완성 카드를 밀어내 핵심 동작을 가리지 않도록, 슬롯은 핵심 콘텐츠보다 시각 우선순위가 낮아야 한다.

## 5. 범위와 비범위 · 리스크

### 5.1 이번에 하는 것 (In Scope)

- 산출물 목록(`ArtifactListScreen`) 진행 카드 구간 1차 광고 슬롯 **인프라(FR-1~FR-8)**.
- 노출/철거/빈도 게이팅 로직(3장 정책 구현).
- 자체 **플레이스홀더 콘텐츠**(자사 안내·제휴 배너) 1차 세트.
- **"광고/Sponsored" 라벨링** 및 정직성·접근성 충족.
- 노출/클릭 **계측 훅 자리**(로컬/노옵 싱크 허용).
- 정당화 카피의 **기획 의도 정의**(실제 문구·치수는 디자이너 확정).

### 5.2 안 하는 것 (Out of Scope) — 단, 확장 seam은 남긴다

- **실제 광고망(AdSense 등) 연동** 및 DOM 오버레이/외부 SDK 적재. (FR-1 슬롯 어댑터 교체 seam만 확보.)
- **결제·구독 시스템**(법적 이슈로 보류).
- **서버 측 광고 송출/타게팅/입찰**. (NFR-4 추상화로 추후 가능.)
- **2차 노출 위치**(제출 전환 구간·열람 로딩 등) 실제 적용. (재사용 seam만.)
- **프리퀀시 캡·개인화·A/B 광고 최적화**. (실광고 단계 과제.)
- **계측 이벤트의 외부 송출/대시보드**. (훅 자리만.)

### 5.3 결정 / 가정 (Decisions / Assumptions)

- **결정:** 1차 슬롯 위치는 "활성 작업 카드 묶음 아래 단일 슬롯"(3.1-a). 근거: 최소 침습 + 청킹/단일목적 원칙.
- **결정:** 실패 작업에는 광고를 노출하지 않는다(3.5). 근거: 엣지 케이스 공감·다크패턴 금지.
- **결정:** 1차 KPI는 노출 계측 커버리지·완주율 비열등성(1.4). 근거: 실광고 단가·트래픽 부재로 수익 절대치 목표는 신뢰도 낮음.
- **가정:** 플레이스홀더 콘텐츠는 자사/제휴 정직 콘텐츠만으로 충분히 채울 수 있다(외부 의존 없이 1차 출시 가능).
- **가정:** 단일 캔버스 환경에서 플레이스홀더는 순수 Compose로 충분하며, 실광고 단계에서만 DOM 오버레이 검토가 필요하다.

### 5.4 리스크

| 리스크 | 영향 | 완화 |
| --- | --- | --- |
| 광고가 "정당화"에도 불구하고 이탈을 유발 | 완주율·신뢰 하락 | 최소 침습 배치·즉시 철거·실패 시 미노출, 도입 전후 지표 비교(비열등성 게이트) |
| 플레이스홀더가 "광고처럼 안 보임"(자사 안내와 혼동) | 라벨 인지 실패·향후 실광고 전환 시 거부감 | "광고/Sponsored" 라벨 강제(FR-6), 자사 안내와 시각 분리 |

| 수익 지표 목표의 조기 설정 | 잘못된 KPI로 의사결정 왜곡 | 1차는 인프라·UX 지표 중심, 수익 KPI는 실광고 연동 후 |
| 접근성/대형 폰트에서 슬롯이 핵심 동작 가림 | 사용성 저하 | NFR-2 게이트, 슬롯 시각 우선순위 하향, 디자이너 레이아웃 검수 |

## 6. UX 플로우

### 6.1 상태 전이와 광고 슬롯 노출/부재 규칙

아래 표가 구현의 "광고 게이팅 진실의 원천"이다. 상태 판정은 `ArtifactListViewModel`의 `renderJobs`와 `artifacts`를 기준으로 한다.

| 화면 상태 | 활성 작업 (PENDING/RUNNING) | 광고 슬롯 | 근거 |
| --- | --- | --- | --- |
| 빈 상태 (`isEmpty=true`) | 없음 | 없음 | EmptyState 맥락; 대기 없음 |
| PENDING 또는 RUNNING 1개 이상 | 있음 | 노출 | 대기 맥락 존재 |
| FAILED-only (활성 작업 전무) | 없음 | 없음 | 실패 공감 우선; 다크패턴 금지 (3.5) |
| PENDING + FAILED 혼재 | 있음 (PENDING) | 노출 | PENDING 1개라도 있으면 대기 맥락 성립 |
| SUCCEEDED 전환 직후 (폴링 반영) | 0으로 감소 | 즉시 철거 | 완성 우선; 비인질 원칙 (3.4) |
| 완성-only (`artifacts`만 존재) | 없음 | 없음 | 대기 맥락 없음 |
| 로딩 중 (`state.loading=true`) | 미확정 | 없음 | 스켈레톤 우선; 슬롯 판단 불가 |

**게이팅 판정 로직 한 줄 요약:** `showAdSlot = renderJobs.any { it.status.isActive }` — `isActive`는 기존 코드의 `GenerationJobStatus.isActive` 프로퍼티(PENDING/RUNNING)를 그대로 사용한다. FAILED가 몇 개든 isActive 작업이 1개 이상이면 노출한다.

## 7. 레이아웃·비주얼 명세

### 7.1 AdSlot 컨테이너

`AdSlot`은 새 최상위 컴포저블이다. `RmCard`를 **재사용하지 않는다** — 진행 카드(`RmCard`)와 시각 위계를 달리해 "광고임이 분명"하게 만들기 위해서다.

**컨테이너 스펙:**

| 속성 | 값 | 토큰 |
| --- | --- | --- |
| 배경 | 흰색 | `colors.surface` (#FFFFFF) |
| 테두리 굵기 | 1dp | — |
| 테두리 색 | slate-200 | `colors.border` (#E2E8F0) |
| 모서리 반경 | 16dp | `RmSpacing.radiusCard` |
| 그림자/Elevation | 0 (무그림자) | — |
| 내부 패딩 (전체) | 16dp | `RmSpacing.space4` |

**RmCard와의 차이점 (시각 위계 분리 근거):**

| 속성 | RmCard (진행·완성 카드) | AdSlot |
| --- | --- | --- |
| 테두리 색 | `borderSubtle` slate-100 (#F1F5F9) | `border` slate-200 (#E2E8F0) — 더 진함 |
| 그림자 | `elevCard` = 1dp (살짝 떠 있는 느낌) | 0dp (페이지 표면에 붙은 느낌) |
| 시각 무게 | 주 콘텐츠 | 보조/부속 콘텐츠 |

진행 카드가 시각적으로 더 높이 "떠 있는" 주 레이어이고, AdSlot은 그 아래 붙은 보조 레이어다. 같은 흰 배경이되 테두리와 그림자 차이로 위계가 자연스럽게 읽힌다.

### 7.2 내부 레이아웃 (Column, 세로 스택)

```
AdSlot(
  bg=surface, border=1dp border, radius=radiusCard, elevation=0, padding=space4
)
│
├── Row(fillMaxWidth, verticalAlignment=CenterVertically)
│   ├── Badge(text="광고", fg=textLabel, bg=borderSubtle)   ← 좌측 상단
│   └── Spacer(Modifier.weight(1f))                        ← 우측 여백 채움
│
├── Spacer(height=space3)                                    ← 12dp
│
├── Box(fillMaxWidth, heightIn(min=88.dp))                   ← placeholder 영역
│   └── [AdContentProvider.Content()]                        ← 주입된 콘텐츠 컴포저블
│
├── Spacer(height=space3)                                    ← 12dp
│
└── Text(                                                    ← 정당화 카피
        text="AI 생성 비용은 광고로 충당돼요. 생성이 끝나면 바로 사라져요.",
        style=RmTextStyles.caption,
        color=colors.textTertiary,
        maxLines=2
    )
```

### 7.3 "광고" Badge 명세

| 속성 | 값 | 근거 |
| --- | --- | --- |
| 컴포넌트 | 기존 `Badge` 재사용 | 신규 컴포넌트 남발 금지 |
| 텍스트 | `"광고"` | 8.2 참조 |
| 전경색 | `colors.textLabel` (#334155, slate-700) | slate-100 배경 위 대비 ≈ 7:1 (WCAG AA 충족) |
| 배경색 | `colors.borderSubtle` (#F1F5F9, slate-100) | 중립 — 유색 상태 배지와 의미적 구분 |
| 위치 | Row 최좌측, 상단 정렬 | 한국어 좌→우 읽기 순서상 슬롯 진입 즉시 인식, 투명성 원칙 강화 |

*유색 상태 배지(primary/warning/danger/success)와 색상을 달리해 "상태"가 아닌 "메타 라벨"임을 즉시 전달한다. 좌측 배치는 업계 관행인 우측 코너 배치 대신 채택했다 — "광고"가 콘텐츠보다 먼저 읽혀야 투명성 원칙이 성립하기 때문이다.*

### 7.4 placeholder 콘텐츠 영역

| 속성 | 값 |
| --- | --- |
| 최소 높이 | 88dp |
| 배경 | 없음 (AdSlot 컨테이너 배경 그대로) |
| 콘텐츠 | `AdContentProvider.Content()` composable lambda |
| 로드 실패 시 | AdSlot 전체를 렌더하지 않음 (7.6 graceful degradation) |

placeholder 내부 표준 구조 (각 변형이 따르는 레이아웃):

```
Row(verticalAlignment=Top, horizontalArrangement=spacedBy(space3))
├── Icon(24dp, textSecondary)
└── Column(verticalArrangement=spacedBy(space1))
    ├── Text(제목, bodyS + Bold, textPrimary)
    ├── Text(설명, caption, textSecondary, maxLines=2)
    └── TextLink(CTA 문구, caption + Bold, primary)
```

### 7.5 AdSlot과 인접 카드의 간격

| 위치 | 간격 | 적용 방법 |
| --- | --- | --- |
| 마지막 진행 카드 ↔ AdSlot 앞 | `space5` (20dp) | Column `spacedBy(space3)` 기본값에 AdSlot 직전 `Spacer(Modifier.height(RmSpacing.space2))` 삽입 (8dp 추가 = 합계 20dp) |
| AdSlot ↔ 첫 번째 완성 카드 | `space3` (12dp) | Column `spacedBy`의 기본값 그대로 |

진행 카드 묶음과 AdSlot 사이를 20dp로 넓혀 시각적 "섹션 구분" 효과를 준다. AdSlot과 완성 카드는 같은 12dp 간격으로 자연스럽게 연결한다.

### 7.6 graceful degradation (콘텐츠 로드 실패)

`AdContentProvider`가 콘텐츠를 반환하지 못하면(null 반환, 예외 등):
- `if (hasContent)` 조건으로 AdSlot 전체를 미렌더.
- 빈 공간, 깨진 박스, 에러 메시지 없음.
- 인접 카드 레이아웃에 영향 없음.
- Spacer(space2) 도 함께 미삽입해 레이아웃 점프 없음.

### 7.7 접근성 포커스 순서

```
[진행 카드 1] → [진행 카드 N] → [완성 카드 1] → [완성 카드 N] → [AdSlot]
```

AdSlot을 포커스 순서 **마지막**에 배치한다. 핵심 동작(진행 확인, 결과 열람, 실패 복구)이 광고보다 먼저 포커스를 받아야 한다(NFR-2, UX 원칙 5). Compose에서는 `Modifier.semantics { traversalIndex = Float.MAX_VALUE }`를 AdSlot에 적용한다.

| --- | --- | --- |

## 9. 수용기준(Acceptance Criteria)

개발팀이 "되었다"를 판정할 수 있는 체크리스트다. 각 항목은 PASS/FAIL로 판정 가능하다.

### 9.1 노출 게이팅 (FR-2)

- [ ] `renderJobs`가 비어 있을 때 AdSlot이 컴포지션에 존재하지 않는다.
- [ ] `renderJobs`가 FAILED 작업만 포함할 때 AdSlot이 컴포지션에 존재하지 않는다.
- [ ] `state.isEmpty = true`일 때 AdSlot이 컴포지션에 존재하지 않는다.
- [ ] `state.loading = true`일 때 AdSlot이 컴포지션에 존재하지 않는다.
- [ ] `artifacts`만 있고 `renderJobs`가 없을 때 AdSlot이 렌더되지 않는다.
- [ ] `renderJobs`에 PENDING 작업이 1개 이상 포함될 때 AdSlot이 렌더된다.
- [ ] `renderJobs`에 RUNNING 작업이 1개 이상 포함될 때 AdSlot이 렌더된다.
- [ ] `renderJobs`에 PENDING 1개 + FAILED 1개가 섞여 있을 때 AdSlot이 렌더된다.

### 9.2 즉시 철거 (FR-3)

- [ ] 폴링으로 마지막 활성 작업이 SUCCEEDED로 전환되어 `renderJobs`에서 사라지면, 다음 컴포지션 사이클에서 AdSlot이 미존재한다.
- [ ] 마지막 활성 작업이 FAILED로 전환되어 FAILED-only 상태가 되면 AdSlot이 사라진다.
- [ ] 완성 결과 카드(`ArtifactSummaryCard`)가 렌더된 화면에 AdSlot이 함께 존재하지 않는다 (활성 작업이 없는 상황 기준).

### 9.3 단일 슬롯 상한 (FR-4)

- [ ] PENDING 작업이 3개, RUNNING 작업이 2개 동시에 있을 때 화면에 AdSlot 인스턴스가 정확히 1개만 존재한다.

### 9.4 광고 라벨링 (FR-6)

- [ ] AdSlot이 렌더되는 모든 상태에서 `Badge(text="광고")`가 시각적으로 보인다.
- [ ] "광고" Badge의 alpha가 1.0이며 불투명하게 표시된다(숨김 처리 불가).
- [ ] Badge 전경(`textLabel` #334155)과 배경(`borderSubtle` #F1F5F9)의 명도 대비가 WCAG AA 기준(≥ 4.5:1)을 만족한다. (참고: slate-700 on slate-100 ≈ 7:1)

### 9.5 통제감·진행 표시 보호 (FR-7)

- [ ] AdSlot이 렌더된 상태에서 `ArtifactJobCard`의 `JobStatusBadge`, 제목 텍스트, `IndeterminateProgressLine`, `CircularProgressIndicator`가 화면에서 가려지지 않는다.
- [ ] AdSlot이 렌더된 상태에서 `ArtifactSummaryCard`의 chevron 클릭이 정상 동작한다.
- [ ] AdSlot이 렌더된 상태에서 FAILED 카드의 "다시 만들기", "경험 다시 고르기", "기록 삭제" 버튼이 클릭 가능하다.
- [ ] AdSlot이 "만들기 시작"·핵심 동작 버튼 중 어느 것도 화면 밖으로 밀어내거나 가리지 않는다.

### 9.6 계측 이벤트 훅 (FR-8)

- [ ] AdSlot이 처음 컴포지션에 진입할 때(LaunchedEffect/DisposableEffect) `onImpression` 콜백이 정확히 1회 호출된다.
- [ ] 같은 생성 세션 동안 폴링에 의해 상태가 갱신되어도 `onImpression`이 중복 호출되지 않는다.
- [ ] placeholder 콘텐츠 내 CTA(TextLink) 클릭 시 `onAdClick` 콜백이 호출된다.
- [ ] `onImpression`, `onAdClick`이 기본값(no-op lambda)으로 설정되어도 크래시가 없다.

### 9.7 교체 가능한 슬롯 인프라 (FR-1 / NFR-4)

- [ ] `AdContentProvider` 인터페이스를 플레이스홀더 구현체 → 테스트용 stub으로 교체해도 `ArtifactListScreen` 호출부 코드 변경이 없다.
- [ ] 새 `AdContentProvider` 구현을 추가할 때 기존 게이팅·라벨링·계측 코드에 수정이 없다.
- [ ] 2차 노출 위치에 AdSlot을 재사용할 때 게이팅 조건을 별도 파라미터로 주입할 수 있다.

### 9.8 graceful degradation (NFR-5)

- [ ] `AdContentProvider.Content()`가 예외를 던지거나 콘텐츠를 제공하지 못할 때 AdSlot이 렌더되지 않으며 화면에 빈 박스나 에러 UI가 남지 않는다.
- [ ] AdSlot 미렌더 시 인접 카드(진행 카드, 완성 카드) 레이아웃이 정상 표시된다.

### 9.9 성능 (NFR-1)

- [ ] AdSlot의 유무와 관계없이 폴링 사이클(3초) 동작에 지연이 없다.
- [ ] AdSlot 콘텐츠 로딩이 메인 컴포지션 스레드를 블록하지 않는다(비동기 로딩 시 코루틴 사용).

### 9.10 접근성 (NFR-2)

- [ ] AdSlot에 `contentDescription = "광고 영역"`이 적용되어 스크린 리더가 읽는다.
- [ ] placeholder 콘텐츠에 콘텐츠 제목을 포함한 `contentDescription`이 제공된다.
- [ ] AdSlot의 포커스 탐색 순서가 진행 카드·완성 카드 뒤에 온다 (`traversalIndex = Float.MAX_VALUE` 설정).
- [ ] OS 최대 글꼴 크기(×1.5 이상)에서 AdSlot이 진행 카드·완성 카드 전체를 화면 밖으로 밀어내지 않는다.

### 9.11 정직성·브랜드 (NFR-3)

- [ ] AdSlot 내 모든 텍스트가 UX 원칙 2(긍정 프레이밍, 대화형 톤)를 준수하며 부정어·죄책감 유발 문구가 없다.

### 9.12 프라이버시 (NFR-6)

- [ ] 1차 플레이스홀더 콘텐츠에 외부 네트워크 요청, 추적 스크립트, 쿠키, PII 수집이 없다.

## 10. 열린 질문 결정 기록

기획 5.5의 3건을 아래와 같이 닫는다. 각 결정은 번복 시 재검토 트리거도 함께 명시한다.

### 10.1 닫기/숨기기 컨트롤 제공 여부 (FR-7)

**결정: 닫기 컨트롤을 제공하지 않는다.**

**선택지 비교:**

| 항목 | 닫기 컨트롤 제공 | 자동 소멸만 (채택) |
| --- | --- | --- |
| 통제감 | 즉시 닫기 가능 ↑ | 완성 시 자동 소멸 → 예측 가능한 통제감 |
| 광고 가치 | impression 이탈 발생 ↑ | impression 유지 |
| 구현 복잡도 | 버튼 + 상태 관리 추가 | 없음 |
| UX 원칙 3 충돌 | 단일 목적 화면에 두 번째 액션 요소 추가 | 충돌 없음 |
| 정당화 전략 일관성 | "닫아야 할 것"이라는 프레이밍 강화 → 2.4 비강압 원칙과 충돌 | 대기 시간 자연 콘텐츠 프레임 유지 |

**근거:** 광고는 이미 세 겹으로 자연 제한되어 있다 — 대기 시간에만(시간), 화면 하단 1슬롯(공간), 진행 카드 아래(위계). 여기에 닫기 버튼을 추가하면 오히려 "광고는 원래 닫고 싶은 것"이라는 부정적 프레이밍을 도입하게 된다. 이는 정당화 전략 2.4(보상형 멘탈모델, 비강압)와 정면으로 충돌한다.

**재검토 트리거:** 실광고 연동 후 완주율이 허용 마진(3%p, 10.2 참조)을 초과하거나, 사용자 피드백에서 광고 거부감이 명시적으로 보고될 경우 닫기 컨트롤 도입을 재검토한다.

### 10.2 완주율 비열등성 허용 마진

**결정: 3%p**

**선택지 비교:**

| 마진 | 의미 | 리스크 |
| --- | --- | --- |
| 1%p | 매우 엄격 | 자연 변동에도 실패 판정, 과검출로 불필요한 롤백 |
| **3%p (채택)** | **현실적 — 비침습 광고의 기대 범위** | 균형 |
| 5%p | 관대 | 실제 UX 저하를 묵인할 위험 |

**근거:** 대기 화면 하단 단일 슬롯 + 즉시 철거 + 실패 미노출 조합이라면 완주율 영향은 3%p 이하로 수렴할 것으로 예상된다. 3%p는 통계적으로 유의미한 저하(실제 사용자 경험 손상)와 자연 변동을 구분하는 합리적 경계다.

**전제 조건 (중요):** 비열등성 판정은 광고 도입 전 2주 이상의 기준선(baseline) 완주율 측정이 선행되어야 한다. **광고 슬롯을 활성화하기 전 기준선 측정 기간을 반드시 확보해야 한다.** 표본 크기: 완주 이벤트 기준 각 그룹(광고 있음/없음) 300건 이상 확보 후 판정한다.
